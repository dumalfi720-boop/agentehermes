📦 Why run Docker

You could install Hermes directly on Ubuntu withpip install hermes-agent. It will work.But it will turn into a nightmareAs soon as you want to run 2 profiles in parallel, update Python or clean uninstall. Docker solves this once and for all.

🎁What Docker delivers to Hermes

  • Isolation:Own filesystem, own dependencies, own .env. Does not pollute the host.
  • Multi-agent:running personnel + marketing + finance side by side, each with different keys.
  • Atomic update: docker compose pull && up -dexchanges the entire image without changing anything on the host.
  • Reproducibility:samedocker-compose.ymlGet the same agent on any VPS.
  • Rollback:went wrong? Returns to the previous image in 30 seconds.

🧠mental model

Image= frozen template, comes from Docker Hub.Container= running image, disposable.Volume= persistent folder outside the container — this is where memory, skills and SQLite live. Updating the image destroys the container, but the volume remains. If you put everything in volume, the update is zero-loss.

⬇️ Install Docker on Ubuntu

The right way in 2026:official Docker script ou Hostinger one-click app. Avoidapt install docker.io— Ubuntu package is usually 6+ months late and almost never includes the compose plugin v2.

📜 Installation via official script

# Conectado como hermes (com sudo)
curl -fsSL https://get.docker.com | sudo sh

# Adicionar usuário ao grupo docker (rodar sem sudo)
sudo usermod -aG docker $USER

# Logout e login pra ativar grupo (ou: newgrp docker)
exit
ssh hermes@SEU_IP

# Verificar instalação
docker --version
docker compose version  # nota: SEM hífen
docker run hello-world

⚠️Compose v1 vs v2 gotcha

  • ✗v1 (discontinued): docker-compose(with hyphen), Python.
  • ✓v2 (current): docker compose(no hyphen), Go, Docker Engine plugin.
  • Old tutorialsthey use v1 and break silently. Always use v2.

💡Hostinger Shortcut

On the Hostinger dashboard, under VPS → Marketplace, "Docker" is one-click. Ubuntu 24.04 + Docker Engine + Compose v2 already configured. Skip the manual steps. Useful for the first time.

📜 docker-compose.yml for Hermes

This is the file that orchestrates everything.Volume bind mountguarantees persistence.env_fileisolate secrets.restart policybrings the agent back after reboot.TZavoids timetable pranks.

📄 minimal docker-compose.yml

services:
  hermes:
    image: nousresearch/hermes-agent:latest
    container_name: hermes
    restart: unless-stopped
    tty: true
    stdin_open: true
    env_file: .env
    environment:
      - TZ=America/Sao_Paulo
    volumes:
      - ~/.hermes:/root/.hermes
    networks:
      - hermes-net

networks:
  hermes-net:
    driver: bridge

🔍 Deciphering line by line

  • image:which version of Hermes to download.:latestalways take the newest one.
  • restart: unless-stopped:it goes up on its own after reboot, but it doesn't restart if you stopped it manually.
  • tty + stdin_open:enabledocker exec -it hermes hermeswith complete TUI.
  • env_file: .env:reads variables from the host's .env into the container.
  • volumes ~/.hermes:/root/.hermes:host folder becomes container folder — persistence.
  • TZ=America/Sao_Paulo:adjusts container timezone (default is UTC).

⚠️Common errors in this file

  • Forget volume:Did you recreate the container? Memory, skills, SQLite — all lost.
  • Relative path on volume: ./hermes:/root/.hermesdepends on the execution directory. Use absolute or~/.
  • Withouttty:the Hermes CLI becomes a log of strings without color or formatting.
  • WithoutTZ:crons fire 3 hours earlier than expected (UTC vs Brasilia).

🚀 First climb: docker compose up

All ready indocker-compose.yml. Time to climb.The first time is where almost everything goes wrong— missing token, volume with wrong permission, .env not mounted. Knowing how to read logs and identify symptoms shortens hours into minutes.

⚡ Climbing sequence

# Na pasta com docker-compose.yml e .env
cd ~/hermes-deploy

# Subir em background (-d = detached)
docker compose up -d

# Acompanhar logs em tempo real (-f = follow)
docker logs -f hermes

# Entrar na CLI interativa do agente
docker exec -it hermes hermes

# Verificar que está rodando
docker ps

✓ Signs of success

  • ✓docker psshow containerhermeswith statusUp X minutes
  • ✓Logs without red errors, message"Agent ready"or similar
  • ✓docker exec -it hermes hermesopens the interactive prompt
  • ✓~/.hermes/on the host there are files (SOUL.md, state.db) created by the container

✗ Problem symptoms

  • ✗"Permission denied"on volume → adjust UID/GID orchown
  • ✗"Missing API key"→ .env is not being read (wrong path or wrong name)
  • ✗Container restarts in loop → seedocker logs hermes --tail 50
  • ✗"image not found"→ typo in image name or non-existent tag

🔧 Everyday commands

Five commands cover 90% of operations with Hermes in Docker. Decoration is what separates those who come back to Stack Overflow every time from those who operate the agent in flow.

🛠️ The 5 essential commands

# 1. Subir / atualizar
docker compose up -d

# 2. Reiniciar (após mudar config ou .env do container)
docker compose restart

# 3. CLI interativa do agente
docker exec -it hermes hermes

# 4. Setar config dentro do container (escreve em /root/.hermes/.env)
docker exec -it hermes hermes config set TELEGRAM_TOKEN xxx

# 5. Ver logs
docker logs hermes --tail 100
docker logs -f hermes  # follow modo

📚 Useful secondary commands

  • docker compose pull— download new image without uploading
  • docker compose down— for everything (preserves volumes!)
  • docker compose ps— status of each service
  • docker compose logs hermes --since 1h— last hour logs
  • docker stats hermes— real-time CPU and memory usage
  • docker exec hermes env | grep TELEGRAM— check loaded variables

🔥Dangerous commands – be careful

  • docker compose down -v— erases VOLUMES together. Goodbye memory, skills, SQLite. NEVER in production.
  • docker system prune -a— deletes ALL unused images. Be careful if you have other projects.
  • docker rm -f hermes— removes the forced container. Volume survives, but confuses compose.

🔄 Update without losing data

Hermes releases frequent releases (v0.13.0 on May 7, 2026, "The Tenacity Release"). You needupdate with confidence. The correct flow preserves everything that matters — memory, skills, SQLite, configs — because it's on the volume.

🔁 Secure update flow

# 1. Backup do volume (sempre antes de atualizar)
tar czf hermes-backup-$(date +%Y%m%d).tgz ~/.hermes

# 2. Baixar imagem nova
docker compose pull

# 3. Recriar container com nova imagem
docker compose up -d

# 4. Verificar versão
docker exec hermes hermes --version

# 5. Conferir logs por 1-2 minutos
docker logs -f hermes

# Se algo deu errado, voltar para tag específica:
# editar docker-compose.yml: image: nousresearch/hermes-agent:0.12.5
# docker compose up -d

✓Why is this flow safe

  • Backup tar:30 seconds, saves your life in rollback.
  • Pull before up:image downloaded successfully before moving the current container.
  • Preserved volume: ~/.hermesis not touched when the container is recreated.
  • Easy Rollback:change tag to previous version +up -deverything comes back.

💡Pinning tip

In critical production, avoid:latest. Use explicit tag (:0.13.0). You decide when to update and have deterministic rollback.:latestIt's convenient for exploration, but it can break unexpectedly when a major version comes out with a breaking change.
