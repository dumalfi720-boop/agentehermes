💾 Why Connect GitHub

Hermes accumulates value over time:months of feedback, dozens of skills created, rich memory, refined settings. Losing this due to a hardware failure or adocker compose down -vwrong is traumatic. Git solves it with zero operational effort after setup.

🎁What do you get

  • Full backup:SKILL.md, MEMORY.md, USER.md, SOUL.md, config.yaml versioned
  • Resilience:VPS caught fire?git cloneon another server, restore in minutes
  • History:see the evolution of your skills over months, roll back
  • Sharing:multiple agents point to the same skills repo
  • Confidence to try:Did you test a new skill and break everything?git checkoutback

🧠mental model

Private repo= no one sees (even if username known).You= single developer + single agent user.Commit message= "snapshot of day X".Automatic push via cron= continuous backup without you remembering.

TOPIC 2
2

🔐Personal Access Token (PAT)

GitHub retired password authentication in 2021. Today it usesPersonal Access Token. There are two types:classic(older, broad scopes) andfine-grained(newer, surgical scope).Always prefer fine-grained.

📍 Where to create

GitHub → Foto perfil (canto superior direito)
       → Settings
       → Developer settings (rolar até o fim do menu lateral)
       → Personal access tokens
       → Fine-grained tokens
       → Generate new token

✓ Recommended configuration

  • Expiration:90 days (periodic renewal force review)
  • Repository access:Only select repositories → your Hermes repo
  • Permissions:Contents: Read and write, Metadata: Read
  • Nothing more than that— no Workflows, no Issues, no Actions

✗ Risky setup

  • Classic token withreposcope:access to ALL your repos
  • No expiration:forgetting to revoke is easy
  • "All repositories":scope too broad for 1 agent
  • Workflows + admin:leaked token could affect CI/CD

💡Principle of least privilege

Surgical token limits damage if leaked.Think:“If this token were to hit the network now, what would be the worst case scenario?” With limited fine-grained, the worst thing is that someone changes your repo — recoverable via history. With classicrepo, someone deletes everything.

TOPIC 3
3

🚨 NEVER paste token in agent chat

This is the most common mistake made by those just starting out.Tokens pasted into chat end up in SQLite history, in logs, in searchable sessions.The right way is to usehermes config set, who writes in a safe place with applied self-writing.

❌What NOT to do

# ERRADO — token vai pro histórico em texto puro
"Use o token ghp_xxxxxxxxxxxxxxx pra fazer push"

# ERRADO — log SQLite captura tudo
"Salva esse token na memória: ghp_xxx"

✅The correct way

# 1. Setar via config set (escreve em /root/.hermes/.env, com redação)
docker exec -it hermes hermes config set GITHUB_TOKEN ghp_xxxxxxxxxxxxxxx

# 2. Verificar (mostra valor mascarado)
docker exec hermes hermes config get GITHUB_TOKEN

# 3. Restart pra Hermes reler o .env
docker compose restart

# 4. Agora você pode pedir no chat sem expor o token:
"Faça commit e push do estado atual do agente"

🛡️Self-writing of logs

Hermes automatically redacts known secrets into~/.hermes/logs/errors.log e gateway.log. But this only works for values ​​that ithe knowswhich are secret — environment variables ending in_TOKEN, _KEY, etc. Cheating in free chat does not go through the newsroom.

🚨If you have already committed .env by mistake

  1. Revoke IMMEDIATELYall the tokens/secrets that were there (GitHub Settings → revoke)
  2. Generate new tokens with surgical scope
  3. Update~/.hermes/.envwith the new ones viahermes config set
  4. Usegit filter-repoor BFG to remove from history
  5. Force push (caution):git push --force-with-lease

But know:If it went to public repo even for 1 minute, scraping bots have already captured it.Prevention > remediation.

🤝 Ask Hermes to configure everything

Direct application of the T1.2 mindset:ask Hermes himself. The skillgithubbundled already knows how to create repo via API, generate.gitignorecorrect, commit first. You don't need to memorize git commands.

💬 Full prompt via Telegram

"Configure this agent as private repo on GitHub namedhermes-personal-backup. Create .gitignore by blocking .env, .env.*, state.db, state.db-*, logs/, *.log, permissions.json, *.key, *.pem. ✓What will happen

  1. Hermes charges skillgithubvia progressive disclosure
  2. Create private repo via GitHub API using your token
  3. Generates.gitignoreincluding the patterns you requested
  4. It will ask permission to rungit init, git commit, git push— approve withsession
  5. Give you the repo link

⚠️Before approving "always" permission

Review the.gitignorethat he proposed. If something is missing (e.g. he forgotauth_cache/), ask to add. Just checkalways allowforgit pushwhen the setup is validated and you are sure that.envis blocked. A backup cron only works withalways allow— but it's worth testing manually first.

📂 container's .env vs host's .env

They existTWO .env filesand confusing them is a classic source of wasted hours debugging. Hermes reads the container. Edit the host = change has no effect.

📍 Both .env

FileWho readsWhen it applies
~/hermes-deploy/.env(host)docker composeContainer uplift, inject initial env vars
/root/.hermes/.env(container)Hermes at runtimeWhenever you need TOKEN, KEY, etc.

✏️ Edit/delete key

# Forma recomendada — config set
docker exec -it hermes hermes config set GITHUB_TOKEN ghp_novo
docker compose restart

# Edição manual (se precisar)
docker exec -it hermes nano /root/.hermes/.env
# remove ou altera linha
# salva, sai (Ctrl+O, Ctrl+X)
docker compose restart

# Ver o que está ativo
docker exec hermes hermes config get GITHUB_TOKEN  # mascarado
docker exec hermes env | grep GITHUB

🤔Typical symptom of confusion

"I changed the token in.envbut Hermes keeps using the old one." You edited the.envwrong (from the host) or forgot to dodocker compose restart. Always:config set + restart, oredit in container + restart. Forgetting to restart is the #2 most common mistake.

💡mnemonic rule

host.env= recipe for lifting container. Stir rarely.Container .env= agent configuration at runtime. Always stirconfig set. Alwaysrestartafter any changes.
