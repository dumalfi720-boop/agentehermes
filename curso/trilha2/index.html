☁️ Hiring and configuring VPS

Why VPS, which provider to choose, which plan fits, which OS to install and how to make the server secure before uploading Hermes.

What is it:

VPS is a dedicated virtual machine in the cloud with a fixed public IP, connected 24 hours a day. Unlike a laptop, it doesn't sleep, it doesn't fall down when you close the lid, and it doesn't depend on your home Wi-Fi.

Why learn:

Crons are only reliable on an always-on machine. Telegram needs the gateway to be accessible continuously. Your personal machine was not made for this. Hostinger's KVM 2 costs the equivalent of Spotify Premium and fits multiple agents.

Key concepts:

VPS = Virtual Private Server. KVM = virtualization type (kernel-level, more isolated than LXC). NVMe = fast disk. Monthly bandwidth = transfer limit (8TB on KVM 2 is generous for Hermes).

What is it:

Hostinger has the best onboarding for Brazilians (price in reais, PT-BR support, simple panel). Hetzner is the cheapest for shipping. DigitalOcean has a rich ecosystem of tutorials. Oracle Always Free gives free micro-VPS.

Why learn:

The wrong choice turns into friction for months. Hostinger is the recommended route to get to Hermes first — simple installation, fast NVMe, and significant welcome discount. Hetzner demands more technical autonomy.

Key concepts:

Hostinger: $8.99/month discounted, $14.99 renewal. Hetzner: ~€4 per equivalent. DO: $12+/month. Oracle Free: ARM64 with free 24GB RAM, but high registration bureaucracy.

What is it:

Hostinger's KVM 2 offers 2 vCPUs, 8 GB RAM, 100 GB NVMe and 8 TB bandwidth. It fits Hermes + Docker + a couple of heavy skills (browser automation, vision) without breaking a sweat.

Why learn:

Undersize (KVM 1, 4GB) makes Docker choke when you ask for something that loads Chromium. Oversizing (KVM 4) is only valid if you are going to run multiple profiles or multi-agent kanban.

Key concepts:

Table: KVM 1 ($6.49) is OK for testing; KVM 2 ($8.99) is the sweet-spot; KVM 4 ($12.99) for multiple agents; KVM 8 ($25.99) for Kanban with multiple parallel workers.

What is it:

Ubuntu 24.04 LTS is the recommended standard: support until 2029, giant community, all tutorials use it. Choose a memorable hostname (e.g.hermes-pessoal), nearby datacenter (São Paulo if possible), and activate daily backup.

Why learn:

Less popular distros (CentOS, Alpine) have gotchas with Docker, NodeJS and Python. LTS = guaranteed security updates. Automatic backup saves you when you break something at 11pm on a Friday.

Key concepts:

LTS = Long Term Support (5 years). Hostname appears at the prompt — makes it easier to identify the terminal. Nearby datacenter reduces SSH and Telegram latency. Hostinger Backup ~$1/month more, worth every penny.

What is it:

After the server goes up, connect viassh root@SEU_IP. Update the system withapt update && apt upgrade -y. Create a non-root user, install ufw and enable minimal firewall (22 SSH, 443 HTTPS).

Why learn:

Running everything as root is a recipe for disaster. One poorly copied command and you deleted the entire /etc. Separate user limits damage. ufw blocks ports you don't use, reducing attack surface.

Key concepts:

adduser hermes + usermod -aG sudo hermescreates user with escalation.ufw default deny incoming + ufw allow 22 + ufw enableactivates the firewall.

What is it:

Swap password authentication for SSH key (ssh-keygen + ssh-copy-id), disable direct root login, install fail2ban (blocks IPs with brute force attempts).

Why learn:

Public VPS takes thousands of login attempts per day. Without hardening, it's just a matter of time. With key + fail2ban + ufw, the server is practically untouchable for brute-force.

Key concepts:

Edit/etc/ssh/sshd_config: PasswordAuthentication no e PermitRootLogin no. Restart withsystemctl restart sshd. Before closing the session, open another terminal and test key login — if it breaks, you still have access.

🐳 Docker installation + auto-deploy

Why run inside a container, how to upload Hermes via Docker Compose, day-to-day commands and how to update without losing memory or skills.

What is it:

Docker isolates Hermes in a container with its own filesystem, its own Python dependencies, its own .env. You can run 3 Hermes in parallel (personal, marketing, financial) on the same VPS without conflict.

Why learn:

Without Docker, updating Python or libraries becomes a drama. Skills that depend on specific versions fight. When something fails, you don't know if it was the system or the agent. Container is the clean edge.

Key concepts:

Image = template. Container = running instance. Volume = persistent folder outside the container.~/.hermesit has to be volume, otherwise you lose memory when recreating the container.

What is it:

The right way:curl -fsSL https://get.docker.com | sh. This script is maintained by Docker itself and installs updated engine + compose plugin. Hostinger also offers one-click "Docker app" in the dashboard.

Why learn:

Ubuntu Package (apt install docker.io) is usually late. Docker Compose v1 (in Python) is discontinued — only v2 (Go plugin) works. Catching the wrong one costs hours debugging "command not found".

Key concepts:

After installing:usermod -aG docker hermesto run without sudo. Logout + login. Test withdocker run hello-world. Compose v2:docker compose(without hyphen), nodocker-compose.

What is it:

Filedocker-compose.ymldeclares hermes service: official image, volume~/.hermes:/root/.hermes(persistence),env_filepointing to .env, andrestart: unless-stoppedto go up alone after reboot.

Why learn:

Without volume, when you recreate the container you lose all memory, skills, sessions. Without restart policy, after a power outage the agent will not return alone. Without env_file, secrets become hardcoded strings — bad for versioning.

Key concepts:

Volume bind mount = host folder appears inside the container.env_file: .envinjects variables.TZ=America/Sao_Pauloadjust timezone.tty: trueleaves the interactive agent viadocker exec.

What is it:

Command:docker compose up -drises in the background.docker logs -f hermesshows the log in real time.docker exec -it hermes hermesenters the agent's interactive CLI.

Why learn:

The first climb is where almost everything goes wrong: wrong token, volume with wrong permission, .env not mounted. Knowing how to read logs and identify the symptom helps you fix it in minutes instead of hours.

Key concepts:

up -d= detached.logs -f=follow.exec -it= interactive + tty. Common mistakes:"permission denied"on volume → adjust uid/gid;"missing API key"→ Wrong .env.

What is it:

The 5 commands you will use 90% of the time:docker compose up -d, docker compose restart, docker exec -it hermes hermes, docker exec -it hermes hermes config set X=valor, docker logs hermes --tail 100.

Why learn:

Memorizing these 5 commands eliminates tutorial dependency. You become autonomous to debug and adjust settings in seconds.hermes config setwrites to the container's .env — not to be confused with the host's .env.

Key concepts:

After config set, alwaysdocker compose restartfor the agent to reread.docker compose pulldownload new image.docker compose downfor everything (preserves volumes).

What is it:

Safe Flow:docker compose pulldownload new image,docker compose up -drecreates container with the new version. The volume~/.hermesremains untouched — memory, skills, state.db all there.

Why learn:

Hermes releases frequent versions (v0.13.0 came out 3 days ago). You need to upgrade with confidence. Confusedown -v(deletes volume) withdown(preserves) destroys months of work. Decorating is safe.

Key concepts:

Before any update: volume backup (tar czf hermes-backup.tgz ~/.hermes). docker compose down= for;down -v= to + erase volumes. NEVER use-von an agent in production.

🤖 Inference and Model Provider

Codex OAuth, OpenRouter, Anthropic, MiniMax, local models. How to decide between cost, privacy, quality and volume.

What is it:

Codex OAuth uses your ChatGPT account (Plus $20 or Pro $200) as the provider. Login via browser, without API key. Standard Model:gpt-5.3-codexwith vision. All within OpenAI's fixed plan.

Why learn:

It is the cheapest option for intense use. Using the API, 10 long conversations a day easily cost $100/month. For Plus, the same usage costs a flat $20. Pro releases o3, o4-pro and GPT-5 with high limits.

Key concepts:

Configure withprovider: "codex"in config.yaml. Hermes asks you to open the URL in the browser, you log in, it captures the token. Limitation: ChatGPT plan rate limits — not “truly unlimited”.

What is it:

OpenRouter is an aggregator: you put in credit, choose any model (Claude, Gemini, DeepSeek, Grok, GPT) and pay by token.openrouter/owl-alphaIt's free. Gemini Flash costs ~$0.075/1M tokens.

Why learn:

Allows you to change model per task. Context compression can use Gemini Flash (cheap). Main conversation uses Claude. Watchdog can use owl-alpha (free). You adjust each slot to your budget.

Key concepts:

Configure withOPENROUTER_API_KEY + provider: "openrouter". New models in v0.13.0:deepseek/deepseek-v4-pro, x-ai/grok-4.3, tencent/hy3-preview.

What is it:

Claude Opus / Sonnet via Anthropic's direct API or via GMI (reseller). The model mentioned in the Hermes v0.13.0 examples isclaude-opus-4.6. Excellent at complex code and long reasoning.

Why learn:

When the task is critical (code review, long writing, strategic planning), Claude has superior quality. It's worth the extra cost for occasional use. For high volume, Codex+OpenRouter is more economical.

Key concepts:

Anthropic is not natively unenforced in Hermes — a natively trusted model. Tools used with native confidence. Use GMI for centralized billing if you don't already have an Anthropic account.

What is it:

MiniMax OAuth: browser login, without API key (similar to Codex). Nous Portal:hermes auth + provider: "nous". Location: any OpenAI-compatible endpoint (Ollama, LM Studio, vLLM) withbase_url: "http://localhost:1234/v1".

Why learn:

Extreme privacy scenarios (sensitive data never leaves the network), disconnected use, zero cost for infinite volume (after hardware). Mac Mini M2 with Qwen or LLaMA runs 90% of personal tasks.

Key concepts:

Models without enforcement (trusted): Claude, DeepSeek, Qwen, LLaMA. Models with automatic enforcement: gpt, codex, gemini, gemma, grok. Location requires hardware: 16GB+ for the 7B model, 32GB+ for the 13B.

What is it:

Top picks from release v0.13.0:gpt-5.3-codex(Codex OAuth standard),deepseek/deepseek-v4-pro(strong and cheap reasoning),x-ai/grok-4.3(long answer),claude-opus-4.6(premium quality),MiniMax-M2.7(without API key).

Why learn:

The right model on the right task changes the experience 10x. Codex for general, DeepSeek for mathematical reasoning, Claude for long text, Gemini Flash for compression. Mixing is the optimized way.

Key concepts:

Vision: gpt-5.3-codex and Claude Opus support. Tool use enforcement: gpt/codex/gemini/gemma/grok receive automatic enforcement; Claude/DeepSeek/Qwen/LLaMA are natively trusted.

What is it:

Decision tree: already have ChatGPT Plus? Codex OAuth. Want flexibility between models? OpenRouter. Extreme privacy? Location via Ollama. Top-quality tasks? Anthropic. Cheap high volume? OpenRouter + DeepSeek/Gemini.

Why learn:

Without criteria, you keep changing providers all the time. With discretion, you choose once and optimize over time. Switching is trivial (one line in config.yaml), but jumping for no reason burns energy.

Key concepts:

Initial recommendation: Codex OAuth as main + OpenRouter as fallback (Gemini Flash for compression). High privacy? Adds local endpoint. Migration between providers: zero work — just edit config.yaml + restart.

📱 Telegram: BotFather + authorization

Why start with Telegram, how to create the bot, get your user ID, configure the gateway and debug if the first conversation does not respond.

What is it:

Telegram is the lowest friction messaging method for creating a bot. BotFather is an official bot from Telegram itself that creates other bots in 30 seconds via chat commands. No developer portal, no approval.

Why learn:

Discord requires application registration. WhatsApp goes through a paid gateway (Whapi) or Meta Business Manager. iMessage requires BlueBubbles on a Mac. Telegram solves it in 5 minutes — you validate the complete setup before getting complicated.

Key concepts:

Hermes supports 20+ platforms, but Telegram is the path of least resistance. Voice notes, slash commands, attachments, stickers — everything works. App has solid desktop and mobile client.

What is it:

Open Telegram, search@BotFather. Send/newbot. It asks for the name (display) and username (ending in "bot"). Returns a type token1234567890:AAH...— that's the secret of your bot.

Why learn:

The token is the bot's complete credential — whoever has it controls it. Treating it as a password is critical. Never paste in public chat, never commit to repository. Usehermes config set TELEGRAM_TOKEN xxxinside the container.

Key concepts:

Username must be unique globally on Telegram. Useful customizations in BotFather:/setdescription, /setuserpic, /setcommands(list of slash commands). You can revoke token with/revokeif it leaks.

What is it:

Hermes needs to know who is authorized to speak to him. It's not enough to have the bot — anyone who discovers the username could send a message. To use@userinfobotreturns your numerical user ID (ex:123456789).

Why learn:

Without authorized users configured, the bot either accepts anyone (dangerous) or ignores you (frustrating). User ID is the stable way to authorize — username can change, ID cannot.

Key concepts:

User ID is numeric, unique, permanent. You can authorize multiple IDs if you share the agent with a spouse/partner. For groups, the ID starts with a negative sign (-100...).

What is it:

Inside the container:hermes gateway setupopens a wizard. Paste the bot token, paste your user ID, choose the home channel (default channel for cron deliveries). It saves in~/.hermes/.env e ~/.hermes/config.yaml.

Why learn:

Trying to edit files manually often breaks formatting. The wizard validates each step and tests the connection before saving.hermes gateway startclimb the gateway afterwards.

Key concepts:

Home channel = default channel/chat for automatic deliveries (crons). It can be DM with you or dedicated group. Variables:TELEGRAM_TOKEN, TELEGRAM_HOME_CHANNEL, TELEGRAM_AUTHORIZED_USERS.

What is it:

Send"Hello"for the bot. Expected response: the agent responds by introducing himself. If you don't respond:hermes gateway statusshows status,tail -f ~/.hermes/logs/gateway.logshows error in real time.

Why learn:

First message is the most important test. If it passes, the entire setup is OK. If it fails, the error in the logs is objective (wrong token, unauthorized user, no internet). Solving is literally following the error.

Key concepts:

Frequent errors:"401 Unauthorized"= wrong token;"user not authorized"= ID is not inTELEGRAM_AUTHORIZED_USERS; "connection timeout"= firewall blocking output from the container to api.telegram.org.

What is it:

Voice notes are understood (Telegram sends audio, Hermes transcribes and responds). slash commands/cron, /usage, /compress, /skillswork directly. Permissions: every sensitive action asksallow once / session / always.

Why learn:

Voice messaging greatly speeds up communication on the move (in the car, walking). Slash commands become your remote control: see scheduled crons, compress context, list skills, see token expenditure.

Key concepts:

Permissions persist across~/.hermes/permissions.json. Alwaysonly for reliable things (git push in your repo).Sessionfor exploration.Oncefor single destructive action.

🐙 GitHub: token + .gitignore + hermes config

How to securely connect GitHub, create tight scope token, avoid secret commit and where the agent's .env actually is.

What is it:

GitHub becomes the official backup of your agent: SKILL.md, MEMORY.md, USER.md, SOUL.md, config.yaml versioned in a private repo. If your VPS catches fire, you can restore everything to another server in minutes.

Why learn:

Hermes builds value over time: months of feedback, dozens of skills created, rich memory. Losing this due to a hardware failure is trauma. Git resolves with 0 operational effort after setup.

Key concepts:

Private repo = no one sees it. Skills travel between machines via clone. Multiple agents point to the same shared skills repo. You can even mount the repo in~/.hermes/skillsviagit clone.

What is it:

On GitHub: Settings → Developer settings → Personal access tokens. Fine-grained PAT allows limiting to specific repos with surgical scopes (push/pull only, no delete). Classic PAT is older, broad scopes.

Why learn:

Principle of least privilege. Classic token withrepoallows you to delete all your repos if it leaks. Fine-grained limited to 1 repo + minimum permissions reduces blast radius drastically.

Key concepts:

Token Expiration: Set expiration (90 days is reasonable). Suggested scopes for Hermes:Contents: Read and write, Metadata: Read. Nothing beyond that. When it expires, generate new and update.

What is it:

Right way:docker exec -it hermes hermes config set GITHUB_TOKEN ghp_xxx. This writes in~/.hermes/.envinside the container, with auto-redaction applied to the logs.

Why learn:

Tokens pasted into chat end up in SQLite history, in unredacted logs, in searchable sessions.config setIt is the only way that puts it in a safe place with log protection.

Key concepts:

Hermes has self-written secrets in~/.hermes/logs/errors.log. config setit is the only method guaranteed to pass the newsroom. Paste directly into chat = logs in plain text.

What is it:

Before anygit add ., create.gitignorewith:.env, .env.*, state.db, state.db-*, logs/, *.key, *.pem, permissions.json.

Why learn:

Accidental commit of .env is the most common error on personal VPS. If it was for public repo, the token leaks in seconds (scraping bots). Even in private, history is permanent — whoever enters the repo sees it.

Key concepts:

If you have already committed .env: revoke the token IMMEDIATELY on GitHub and regenerate.git filter-repoor BFG remove from history, but the secret is already leaked. Prevention > remediation.

What is it:

With the token configured, send it on Telegram:"Set up this agent as private repo on GitHub. Create .gitignore by locking .env, state.db, logs and permissions.json. Commit with SKILL, MEMORY, USER, SOUL, config.yaml first."

Why learn:

Direct application of the T1.2 mindset: ask Hermes himself. The skillgithubbundled already knows how to create repo, generate correct gitignore and do initial push — you don't need to memorize git commands.

Key concepts:

Before approving the push, review the .gitignore it proposed. If something is missing, correct it. Markalways allowonly after it validates that it works.

What is it:

There are TWO .env: the host's (~/hermes-deploy/.env, read by docker compose) and the container (/root/.hermes/.env, read by Hermes). The agent reads the second. Edit withdocker exec -it hermes nano /root/.hermes/.env.

Why learn:

Edit the wrong = change has no effect. You restart thinking you will get a new key, but the agent continues using the old one. Confusing the two is a classic source of wasted hours debugging.

Key concepts:

After editing the container's .env:docker compose restart. To delete a key: edit and remove the line.hermes config get GITHUB_TOKENshows the current value (with mask) to confirm which is active.

🌙 First cron: automatic daily backup

Why this is the ideal cron to start with, natural syntax vs CLI, timezone, persistent permissions and how to check that it ran.

What is it:

Daily backup for GitHub is the lowest risk and highest value cron. He just doesgit add . && git commit -m "..." && git pushinside~/.hermes. It doesn't touch the system, it doesn't call risky external APIs, it doesn't spend expensive tokens.

Why learn:

It's the right way to test your cron system with confidence. If it breaks, nothing serious happens. If it works, you gain permanent resilience. It's Hermes' pragmatic "Hello World".

Key concepts:

Principle: start with the most useful and safe cron. Ambitious crons on day 1 (responding to email, posting content) are fragile and dangerous. Backup is trivial and productive.

What is it:

In Telegram chat:/cron add "every 1d at 00:00" "Faça commit e push das mudanças do agente pro GitHub". Hermes interprets natural language, generates the job and confirms scheduling.

Why learn:

Fastest way to create cron. You don't need to memorize classic cron syntax (0 0 * * *). Natural language is translated into Hermes' internal structure (jobs.json).

Key concepts:

Accepted:"every 30m", "every 2h", "every 1d at 09:00", "every monday at 08:00", "weekly". /cron listshows all jobs./cron remove IDoff.

What is it:

CLI equivalent:hermes cron create "every 1d at 00:00" "Backup do agente" --workdir /root/.hermes. Runs inside the container. More explicit, ideal for idempotent provisioning scripts.

Why learn:

When you replicate the setup on another VPS, CLI commands are reproducible. Can version as bootstrap.sh. Telegram is better for exploration; CLI is better for automation.

Key concepts:

Main flags:--workdir(sets cwd, loads AGENTS.md from there),--no-agent(pure script without LLM),--deliver telegram(channel),--skill nome(injects skill).--name backup-diariomakes it easier to identify.

What is it:

Docker Container runs in UTC by default."every 1d at 00:00"it fires at midnight UTC, which is 9pm in Brasilia (UTC-3). To fix:TZ=America/Sao_Pauloin the env vars of the container ORtimezone: "America/Sao_Paulo"in config.yaml.

Why learn:

Cron running 3 hours earlier than expected is guaranteed confusion."Backup at 00:00"which fires at 9pm the day before messes up perception and logs. Setting timezone right at the beginning saves a lot of rework.

Key concepts:

IANA timezone string:America/Sao_Paulo, America/Bahia, Europe/Lisbon. After configuring, restart the container and check withdocker exec hermes date.

What is it:

On the first run, Hermes will ask for permission to rungit commit e git push. Markalways allow— is a safe, recurring action, with scope locked by the fine-grained PAT.

Why learn:

Withoutalways allow, every daily cron hangs waiting for you to respond to the prompt — defeating the purpose of being automatic. Checking once resolves it permanently.

Key concepts:

Persistence in~/.hermes/permissions.json. You can list approved permissions withhermes permissions list. Revoke one withhermes permissions revoke ID. Audit regularly.

What is it:

After creating:/cron listshow job +next_run_at. In the first shot, Hermes responds on the home channel with results. Check on GitHub: new commit appears with message from the agent.

Why learn:

"Thinking that it ran" is different from "having run". Confirming with fact — commit visible on GitHub — closes the loop. If it didn't run, logs in~/.hermes/logs/cron.logshow exactly what failed.

Key concepts:

Lock file~/.hermes/cron/.tick.lockprevents double-run.~/.hermes/cron/jobs.jsonhas the entire state. Execution history is in the SQLite session — searchable viasession_search.
