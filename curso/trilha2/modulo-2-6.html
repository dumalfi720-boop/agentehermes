🛡️ Why is this the first cron

Daily backup for GitHub is the cron oflower risk and higher valuethat you can create on day 1. It only doesgit add && commit && pushinside~/.hermes. It doesn't touch the system, it doesn't call risky external APIs, it doesn't spend expensive tokens, it has no side effects other than uploading a commit.

⚖️Why NOT start with ambitious cron

  • "Automatically reply to email"→ high risk of sending wrong message before you trust
  • "Post content to Twitter/IG"→ public impact if you get the tone wrong
  • "Pay bill/schedule transfer"→ finance deserves a lot of manual validation
  • "Backup to GitHub"→ purely useful, zero side effects, failure is silent

✓Progression principle

Starting with the most useful and secure cron tests the entire system (gateway → AIAgent → tools → permissions → delivery) without putting anything important at risk. If it works, you gain real confidence and protection. If it breaks, you lose nothing but debugging time.

💬 Natural syntax via Telegram

The fastest way to create a cron:slash command on Telegram. You don't need to memorize the classic cron syntax (0 0 * * *) — Hermes interprets natural language and generates the structured job.

📤 Complete command

/cron add "every 1d at 00:00" "Faça commit com timestamp e push das mudanças do agente para o GitHub. Use mensagem 'Backup automático {data}'. Se não houver mudanças, apenas reporte 'Sem alterações'."

🕐 Accepted syntaxes

  • "every 30m"— every 30 minutes
  • "every 2h"— every 2 hours
  • "every 1d at 09:00"— daily at 9am
  • "every 1d at 00:00"— daily midnight
  • "every monday at 08:00"— every Monday at 8am
  • "weekly"— once a week

🔧Related commands

  • /cron list— all scheduled jobs, with next_run_at
  • /cron remove ID— delete a job
  • /cron run ID— force execution now (useful for testing)
  • /cron pause ID / /cron resume ID— temporarily pause
TOPIC 3

⌨️ CLI syntax: hermes cron create

CLI equivalent of cron creation. More explicit than the slash command — ideal for an idempotent provisioning script when you replicate the setup on another VPS. You can version it likebootstrap.shand run in new environments.

⚡ Equivalent command

docker exec -it hermes hermes cron create \
  "every 1d at 00:00" \
  "Backup do agente: commit + push das mudanças" \
  --workdir /root/.hermes \
  --deliver telegram \
  --name backup-diario

🚩 Main Flags

FlagFunction
--workdir <path>job cwd; load AGENTS.md/CLAUDE.md from here
--deliver <plat>destination: telegram, discord, all, origin
--skill <name>injects skill at the beginning of the fresh session
--no-agentscript-only without LLM (for watchdogs)
--script <path>(with --no-agent) .sh or .py script to run
--name <str>identifier readable to you (default: generated)

💡Telegram vs CLI: when to use which

Telegram= interactive exploration, immediate feedback, quick adjustment.CLI= automation, replication, bootstrap scripts. To create 1 cron, Telegram is faster. To upload 10 crons to a new VPS, CLI on a.shis deterministic.

🌍 Timezone: UTC vs local

Classic prank:Docker container runs in UTC by default."every 1d at 00:00"fires at midnight UTC, which is9pm the day beforein Brasília (UTC-3). Cron running 3 hours earlier than expected destroys perception and messes up logs.

🛠️ Two ways to configure

# Opção 1: env var no docker-compose.yml (preferida)
services:
  hermes:
    environment:
      - TZ=America/Sao_Paulo

# Opção 2: config.yaml dentro de ~/.hermes/
timezone: "America/Sao_Paulo"

# Após mudar, restart
docker compose restart

# Verificar timezone ativo
docker exec hermes date
# Esperado: "Sun May 10 18:32:00 -03 2026" (não UTC)

🗺️IANA timezone common strings

  • America/Sao_Paulo— Southeast Brazil (UTC-3)
  • America/Bahia— Bahia (UTC-3, same offset)
  • America/Manaus— Amazonas (UTC-4)
  • Europe/Lisbon— Portugal (UTC+0/+1 with DST)
  • UTC— Greenwich meridian (default container)

⚠️Typical symptom

"I set the cron for midnight but it ran at 9pm." Container still in UTC. Hermes injects the system timezone into the system prompt — without the right TZ, it "thinks" it is 3 hours ahead. Setting timezone right at the beginning saves a lot of rework.

TOPIC 5

✅ Permissions: always allow git push

On the first run of the cron, Hermes will ask for permission to rungit commit e git push. Check "always allow"— is a safe, recurring action, with scope locked by the fine-grained PAT of module 2.5.

🤝Why trust this permission

  • Surgical Token:only allows you to touch 1 repo (the backup one)
  • Reversible Operation:commits can always be rolled back withgit revert
  • .gitignore protects secrets:even if strange commit, .env doesn't go along
  • Audit trail:each push is registered on GitHub with a timestamp
  • Without "always":daily cron hangs waiting for you to respond — breaks the purpose

🔍 Audit permissions periodically

# Listar permissões aprovadas
docker exec hermes hermes permissions list

# Revogar uma específica
docker exec hermes hermes permissions revoke ID

# Arquivo bruto (raramente preciso)
docker exec hermes cat /root/.hermes/permissions.json

📋Monthly audit checklist

  • List permissions and revoke those you don't recognize
  • Confirm tokens (GitHub, OpenRouter) are still active and scoped correctly
  • Look at last push on GitHub — ever without a backup?
  • To check~/.hermes/logs/due to recurring errors

🔍 Check: /cron list, logs, GitHub

"Thinking that it ran" is different from "having run".Confirming with fact closes the loop.Three checkpoints:/cron list(show next run), logs (show execution), GitHub (show actual commit).

✓ Verification checklist

# 1. Cron está agendado?
/cron list   # via Telegram, ou:
docker exec hermes hermes cron list

# 2. Forçar execução agora pra testar (não esperar 24h)
docker exec hermes hermes cron run backup-diario

# 3. Ver logs do tick
docker exec hermes tail -50 /root/.hermes/logs/cron.log

# 4. Conferir commit real no GitHub
# Abre o repo no browser, vê commits do dia

# 5. Buscar histórico via session_search
"Mostre as últimas execuções do cron de backup"

📁Where is the cron state

  • ~/.hermes/cron/jobs.json— all jobs with next_run_at
  • ~/.hermes/cron/.tick.lock— lockfile prevents double-run
  • ~/.hermes/logs/cron.log— executions, errors, exit codes
  • ~/.hermes/state.db— SQLite sessions with history of each cron

🔧If it didn't run at the expected time

  • Container stopped: docker ps; if off,docker compose up -d
  • Wrong timezone: docker exec hermes date, adjust TZ if UTC
  • Lock file locked:to switch off~/.hermes/cron/.tick.lockand restart
  • Permission blocked:seepermissions list, to markalways allow
  • Expired Token:governs PAT on GitHub, update viaconfig set
