MODULE 3.1
3.1~30 min

🧠 Memory: USER.md + MEMORY.md

The two files that the agent injects into the system prompt as frozen snapshots at the beginning of each session. Strict size limits, self-managed via tool.

What is it:

At the start of each session, the contents of USER.md and MEMORY.md are read and injected into the system prompt as a fixed block. Modifications during the session use the toolmemory.

Why learn:

Understanding that it is frozen explains why the agent "forgets" changes made in the middle of the session until the next one. You decide what deserves to persist.

Key concepts:

The files are in~/.hermes/memories/. Snapshot frozen = read once per session. The toolmemoryIt is the official mechanism for changing between sessions.

What is it:

File with your profile: name, role, preferences, communication style, languages, relevant contexts. Hard limit of1,375 characters(~500 tokens).

Why learn:

It's what stops the agent from treating you like a stranger every session. Without useful USER.md, you repeat context every time.

Key concepts:

Focus on stable facts (not current status). Good: "I speak PT-BR, I prefer short answers, I program in Python." Bad: "I'm working on project X right now."

What is it:

The agent's personal notes about the environment: important paths, learned facts, closed decisions, shortcuts. Hard limit of2,200 characters(~800 tokens).

Why learn:

It is the agent's "working" memory. Without it, it rediscovers the environment every time (where repo X is, what stack Y is, how you prefer commits).

Key concepts:

Tight limit forces you to consolidate. When it fills up, the agent itself rewrites older entries to make room. Focus on durable knowledge, not tasks.

What is it:

The toolmemoryhas three operations:add(add entry),replace(to replace),remove(to switch off). The agent uses it alone when you ask "save this".

Why learn:

Knowing that it exists gives you the vocabulary to ask correctly: "save in memory X", "remove entry Y from MEMORY.md", "update what you know about my setup".

Key concepts:

When the file fills up, the agent automatically consolidates — adding redundant entries to free up space. You can audit manually by opening MDs.

What is it:

Each session is recorded in SQLite (~/.hermes/state.db) with full-text index (FTS5). The toolsession_searchallows you to find old conversations by keyword.

Why learn:

It's the "long memory". When something doesn't fit in MEMORY.md but you know it was mentioned, ask the agent to look for it in the session history.

Key concepts:

Memory ≠ history. MEMORY.md is the one that is always loaded; SQLite is the on-demand searchable archive. Backupstate.db= backup of the entire history.

What is it:

Memory is NOT a password vault. It is NOT a to-do list. NOT current status. Saving the wrong type pollutes the snapshot and wastes the 1,375/2,200 char budget.

Why learn:

Poorly used memory becomes noise that impairs reasoning. Secret saved there may leak into remote model logs or prompts.

Key concepts:

Secrets go on~/.hermes/.env. Temporary tasks are left in the conversation. Current status is volatile. Memory = durable knowledge that will serve you in future sessions.

MODULE 3.2
3.2~35 min

🛠️ Skills: SKILL.md + Progressive Disclosure

Reusable procedures in Markdown with YAML front matter. Open standard (agentskills.io). Skills are like functions that the agent learns to call with a name and description.

What is it:

Skill is a SKILL.md file with YAML front matter at the top and instructions below. It tells the agent "when to use this", "how to do it step-by-step", "pitfalls", "how to check".

Why learn:

Skill turns "instruction repeated every session" into "procedure documented once". Each task you have done 3 times becomes a candidate for a skill.

Key concepts:

Skill ≠ code. It is structured natural language. The model reads and follows. Open standard: agentskills.io/specification.

What is it:

YAML header with fields:name, description, version, platforms(optional), andmetadata.hermeswith tags, category, fallback_for_toolsets, requires_toolsets, config.

Why learn:

This is what appears inskills_list(). Bad description = skill never called. Tags and category help you filter and conditionally activate.

Key concepts:

platforms restricts to OS (macos, linux). fallback_for_toolsets activates when another toolset fails. requires_toolsets activates only when the toolset is available. config generates prompts in setup.

What is it:

Four sections:When to Use(triggers),Procedure(steps),Pitfalls(known pitfalls),Verification(how to know it worked).

Why learn:

It is the skeleton that the model expects. Skill without "When to Use" is wrong call. Without "Verification" it is considered successful when it failed.

Key concepts:

Procedure must be numbered and direct. Pitfalls come from experience: what has broken before and how you got around it. Verification is short test of expected output.

What is it:

3-level charging pattern:L0 skills_list()(~3k tokens, just name+description+category),L1 skill_view(name)(full body),L2 skill_view(name, path)(specific reference).

Why learn:

Allows you to have hundreds of skills without overflowing context. The agent only carries a full body when he decides to use it.

Key concepts:

Short and powerful description matters A LOT — that's what goes into L0. Large references are inreferences/and are only pulled at L2.

What is it:

Skill lives in~/.hermes/skills/category/skill-name/. Inside: SKILL.md (required),references/, templates/, scripts/, assets/.

Why learn:

Allows rich skill without inflating main SKILL.md. Scripts are separated for execution, templates for outputs, references for long docs.

Key concepts:

Directory Category is typically the same asmetadata.hermes.category. Versioning the entire skill via git is trivial — it's just Markdown and auxiliary files.

What is it:

Public hub inagentskills.io. Five installation sources:official(bundled),skills-sh(hub),GitHub(user/repo/path),Direct URL, well-known(auto-discovery via /.well-known/agent-skills).

Why learn:

You don't need to create everything from scratch. Watch the demo on X → paste the skill link → the agent installs and uses it.

Key concepts:

Auto-discovery in/.well-known/agent-skillsallows websites to publish official skills — just point the agent to the domain.

MODULE 3.3

👻 Soul: agent identity

SOUL.md occupies slot #1 of the system prompt. It’s identity — who the agent is, how they talk, what they believe. Different from USER.md.

What is it:

SOUL.md is the first block of the system prompt — before memory, before skills. It is the basis of the identity that the model reads for any interaction.

Why learn:

Position matters. Thing in slot #1 has more weight in the model decision. Wrong tone there echoes in everything. The right tone there shapes the entire conversation.

Key concepts:

Always loaded (not conditional). It's the only piece of this trio (SOUL/USER/MEMORY) that has NO documented hard limit, but keep it concise to save tokens.

What is it:

Single file in~/.hermes/SOUL.md. Global scope (applies to any project). Pure Markdown — does not have YAML front matter like SKILL.md.

Why learn:

It's at Hermes' home, not within the project. That's why it doesn't change when you change folders — the agent's identity is constant.

Key concepts:

To have different identities (cheerful marketing, serious finance), you usedifferent Hermes profiles— each profile with its own directory~/.hermes-marketing/, ~/.hermes-financeiro/.

What is it:

Good SOUL.md covers:tone(formal, relaxed, direct),personality(who is the agent),principles(what always/never does),communication rules(language, length, emojis).

Why learn:

It's the difference between "standard ChatGPT" and "agent that looks LIKE you". The more specific, the more characteristic it becomes.

Key concepts:

Avoid internal contradictions. “Be succinct” + “explain each decision in detail” fights. Soul must be coherent as a character.

What is it:

In setups with multiple profiles, soul is what gives each one a distinct personality. Marketing can be light and use emojis. Finance must be sober and direct.

Why learn:

Without their own soul, all profiles sound the same and lose the sense of existing separately. Everyone carries their voice.

Key concepts:

Soul defines voice; skills define capabilities; memory carries facts. The three combine to give character to the profile.

What is it:

"Upgrade your soul to be X" → the agent himself edits SOUL.md. It works surprisingly well because he understands the effect of each line.

Why learn:

You don't need to become a prompt designer. Describe the adjustment in natural language ("be more direct", "fewer emojis", "use practical examples") and the agent formalizes it.

Key concepts:

Always review the diff before accepting. Soul is a critical file — a bad rewrite degrades the entire agent.

What is it:

Fundamental difference:SOUL.md = who the agent is(his identity).USER.md = who you are(your profile). Confusing the two is a common mistake.

Why learn:

Mixing creates strange things. Putting "I'm Nei, programmer" in SOUL.md makes the agent think HE is Nei. Putting "be succinct" in USER.md wastes space.

Key concepts:

Mental test: "If I changed the user, would this still be valid?" If yes → SOUL. If not → USER.

⏰ Crons: scheduling and flags

Crons transform Hermes from reactive to proactive. Session fresh per run, file lock, flags --workdir, --no-agent, --deliver, --skill.

What is it:

Cron = scheduled job. With each trigger, the gateway scheduler opens afresh sessionfrom AIAgent — no previous conversation history, but with SOUL.md, USER.md, and MEMORY.md loaded.

Why learn:

It's what makes the agent truly useful 24/7. You sleep, he works. Wake up to a summary on Telegram.

Key concepts:

Session fresh = prompt should beself-contained. "Continue what you were doing" doesn't work — there's no context. Good: "Audite open PRs in repo X and post summary on Telegram."

What is it:

Chat commands create crons without CLI:/cron add 30m "...", /cron add "every 2h" "...", /cron add "every 1h" "..." --skill blogwatcher.

Why learn:

It's the fastest way to create cron — straight from Telegram. No need to SSH into the server.

Key concepts:

Accepts natural expressions ("every 2h", "every 1d at 09:00") and durations ("30m", "1h", "2d"). Multiple skills with--skill A --skill B.

What is it:

Same capacity via terminal:hermes cron create "every 1d at 09:00" "Audit PRs" --workdir /path/projeto. Ideal for setup and infra-as-code scripts.

Why learn:

CLI allows versioning crons in the repository (scriptsetup-crons.shwhich recreates everything). Reproducible.

Key concepts:

Other useful subcommands:hermes cron list, hermes cron remove <id>. Definition persists in~/.hermes/cron/jobs.json.

What is it:

Sets the fresh session working directory. The scheduler doescdthere before running and detects the project context files: .hermes.md, AGENTS.md or CLAUDE.md.

Why learn:

Cron for a specific project needs the context of that project. Without--workdir, it runs on~and finds nothing.

Key concepts:

Project context priority order:.hermes.md → AGENTS.md → CLAUDE.md → .cursorrules. Only ONE type is loaded per session.

What is it:

Mode that skips the LLM completely. Cron only runs one script.sh/.bash/.py. Empty output = silence. Non-zero output = alert delivered.

Why learn:

Watchdogs (memory, disk, heartbeat) do not need AI. Using LLM there is a waste of tokens. NOAGENT maintains zero costs.

Key concepts:

Default: empty stdout = silent tick, non-zero stderr or exit = alert. Scripts.shrun under bash, others under Python interpreter.

What is it:

--deliver <plataforma>choose destination:telegram, discord, all, origin. --skill <name>injects a skill at the beginning of the fresh session.

Why learn:

Without--deliver, the output goes to the standard channel. With--skill, you guarantee that the right procedure is available even in a fresh session.

Key concepts:

File lock~/.hermes/cron/.tick.lockprevents double-run when the tick delays. You can pass multiple--skillto combine skills.

🔄 Self-Improvement Loop

How the 4 pillars (memory, skills, soul, history) combine in a loop that makes the agent better with use. It’s not magic — it requires feedback.

What is it:

Memory (MEMORY.md/USER.md) = facts. Skills (SKILL.md) = procedures. History (SQLite FTS5) = conversation. The three work together to accumulate useful context.

Why learn:

Knowing which layer to use for which type of learning avoids inflating memory with things that should become a skill, or becoming a skill with something that is just an isolated fact.

Key concepts:

Specific fact → memory. Repeatable procedure → skill. Specific conversation → remains in the history (search via session_search).

What is it:

You ask for a task → it executes → you correct it if wrong → learning is persisted in memory or skill → next session uses this.

Why learn:

It's the rule of the game. Without step 4 (persistence), you will fix the same mistakes over and over again. Everything that evolves the agent goes through this cycle.

Key concepts:

Trigger phrases: "save this in memory", "turns into a skill", "updates your soul", "remember this for next time". Without them, learning disappears at the end of the session.

What is it:

Simple heuristic: make a mistake once, it's carelessness. Got the same thing wrong twice, it's standard — update the corresponding pillar.

Why learn:

Without criteria, you either inflame memory for any detail or ignore what would need to be corrected. 2 is a good pragmatic threshold.

Key concepts:

Error of fact → memory. Process error → skill. Tone error → soul. The right layer depends on the type of error.

What is it:

Every time you find yourself typing "first do X, then Y, watch out for Z" for the 3rd time, that's your cue: become a skill.

Why learn:

Identifying the trigger early saves hours. The skill pays for the investment in just a few executions.

Key concepts:

Ready phrase: "Transform this procedure into a skill called X, with when-to-use, procedure, pitfalls and verification." The agent formalizes.

What is it:

Simple diagnosis: "he spoke in a way I didn't like" → soul. "he forgot fact about my setup" → memory. "he got the step by step wrong" → skill.

Why learn:

Clear routing avoids spending time editing the wrong thing. Thicker Soul doesn't actually fix a lack; Fuller memory doesn't fix tone.

Key concepts:

The mental table: error type → corresponding layer. Applying it wrong degrades everything (memory becoming soul takes too long and wastes the 2,200 chars).

What is it:

Every few weeks, ask: "Show the current contents of SOUL.md, USER.md, MEMORY.md, and the skills list. What is obsolete, contradictory, or redundant?"

Why learn:

Without review, these files become accumulated garbage. Old entries fight with new ones. The agent loses clarity.

Key concepts:

A 10-minute monthly audit is worth more than a rewrite from scratch. Commit the files on GitHub — this way you have a history of the agent’s evolution.
