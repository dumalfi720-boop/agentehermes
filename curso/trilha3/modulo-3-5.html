🔄 Self-Improvement Loop

How the 4 pillars (memory, skills, soul, history) combine in a loop that makes the agent better with use. It's not magic — it requires your systematic intervention.

TOPIC 1

🧬 The 3 layers of learning

The agent accumulates useful context inthree layerswith different natures. Knowing which layer to use for each type of learning is what avoids inflated memory with things that should be skills, and skills created for things that are just facts.

💾 Memory (MEMORY.md / USER.md)

Nature:facts.Loading:always, snapshot frozen at system prompt.

Good for: environment, closed decisions, stable preferences.

🛠️ Skills (SKILL.md)

Nature:procedures.Loading:Progressive Disclosure 3 levels.

Good for: repeatable tasks, flows with clear steps, shareable capacity.

📚 History (SQLite + FTS5)

Nature:conversations.Loading:on demand via session_search.

Good for: specific context that can be useful again. It's not worth promoting memory.

🧭Quick diagnosis

  • • One-off fact("I use PostgreSQL in project X") → memory
  • • Repeatable procedure("how do I deploy X") → skill
  • • Specific conversation("that time we discussed architecture Y") → remains in history
TOPIC 2
2

♻️ Cycle: task → execution → correction → persistence

The complete self-improvement loop hasfive steps. Without the persistence step, you fix the same mistakes over and over again. Without the future use step, persistence is waste.

🔁 The steps

1
Task— you ask the agent something.
2
Execution— he does, based on soul + memory + skills + current context.
3
Correction— you indicate what was wrong or suboptimal.
4
Persistence— learning goes to the right layer (memory, skill or soul).
5
Next session— use what you learned without having to remember it again.

💬Trigger Phrases

Without phrases like this, learning disappears at the end of the session:

  • • "Save this in memory — I always use PostgreSQL for new projects."
  • • "It becomes a skill: deploy project X with these 5 steps."
  • • "Updates your soul to be more direct, without transitional phrases."
  • • "Remember this for next time: my default branch ismain, nomaster."

⚠️No phrase = no persistence

The agent does not decide alone that something has become learning. You need to make it explicit. "I wrote it down mentally" doesn't count — only what went into the archive persists.

TOPIC 3
3

2️⃣ 2 mistakes rule

Simple and powerful heuristics:made a mistake once = carelessness. Made a mistake twice on the same thing = default— update the corresponding pillar. Without criteria, you either inflame memory for any detail or ignore what would need to be corrected.

1st time

Correct and follow. It may have been context specific.

2nd time

Update the right layer. It's standard, not accident.

3rd time

You failed to update. Stop and refresh now.

🎯Why 2 and not 3

3 has already wasted too much time. 1 It is still too early to generalize. 2 is the pragmatic threshold: you have proven the standard without having wasted much time. This is an arbitrary number, but it works as a simple mental rule.

TOPIC 4
4

🪄 Repeat instruction = skill candidate

Every time you find yourself typing"first do X, then Y, be careful with Z"for thethird time, this is the cue:turn skill. Identifying the trigger early saves hours. The skill pays for the investment in just a few executions.

💬Ready sentence

"Transform this procedure into a skill called X. Include When to Use, Step-by-step Procedure, Pitfalls that we have already found and Verification to confirm that it worked. Category: devops."

The agent formalizes with YAML front matter and canonical structure. You just review.

🔍Signs of "this should be skill"

  • •You have a separate step-by-step document (Notion, Google Doc) that you need to open before ordering
  • •Agent makes the same steps wrong in different sessions
  • •You need to explain the procedure to another person on the team
  • •The flow has more than 3 sequential steps
  • •Has known pitfalls that would require explicit warning
TOPIC 5
5

🎚️ Wrong tone = soul; repeated fact = memory; wrong process = skill

Quick diagnosis to choose where to fix. Wrong routing makes the problem worse (thicker soul doesn't fix lack of truth; fuller memory doesn't fix tone).

🗺️ Routing Table

Type of errorLayerExample sentence
Tone/voiceSOUL.md"It's too formal. Update your soul."
Fact about youUSER.md"Save to USER and I'm left."
Environment factMEMORY.md"Saved in memory: backup runs at 3am."
ProcedureSKILL.md"It becomes a skill: deploy project X."
Point contextHistory"When you need to, search the history."

⚠️Wrong routing degrades

  • ✗8-step procedure turning MEMORY.md entry → quickly fills 2,200 chars
  • ✗Tone rules going to USER.md → disappearing in the wrong slot, no power
  • ✗Fact of the environment becoming a skill → skill with 1 phrase in the procedure, without having when to use it
TOPIC 6
6

🩺 Periodic maintenance

Without periodic review, these files becomeaccumulated garbage. Old entries fight with new ones. The agent loses clarity. A 10-minute monthly audit is worth more than rewriting everything from scratch every 6 months.

📅 Review phrase

"Show the current content of SOUL.md, USER.md, MEMORY.md and the list of skills. Point out: what is obsolete, what is contradictory, what is redundant. Propose consolidation."

Suggested cadence

  • • Memory:monthly review
  • • Skills:review every 2-3 months
  • • Soul:review every 1-2 months or when tone bothers you

Version on GitHub

Committee~/.hermes/SOUL.md, memories/ e skills/in a private repo. You get the agent's evolution history, it's easy to revert, you can clone to a new VPS without redoing everything.

📌Complete loop, in one sentence

Use the agent. Correct anything that goes wrong. Have it persist in the right layer. Review periodically. Version everything. In just a few months, the agent is yours — not a generic chatbot.
