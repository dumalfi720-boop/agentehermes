🧠 Memory: USER.md + MEMORY.md

The two files that the agent injects into the system prompt as frozen snapshots at the beginning of each session. Strict limits, managed via tool, complemented by SQLite + FTS5.

🧊 Persistent memory like frozen snapshot

At the beginning of each session, Hermes reads the contents of~/.hermes/memories/USER.md e ~/.hermes/memories/MEMORY.mdand injects everything intosystem promptas a fixed block. This block no longer changes during the session — it is afrozen snapshot.

📐The two files inside

  • USER.md— your profile, ~500 tokens, limit1,375 characters
  • MEMORY.md— agent notes, ~800 tokens, limit2,200 characters
  • Location: ~/.hermes/memories/
  • Loading:once per session (frozen)
  • Inter-session update:via toolmemoryor direct editing

⚠️Practical implication

If the agent updates memoryduringthe current session, the already injected snapshotdoesn't changein this session. Whoever uses this new entry is the next session. That's why "save this in your memory" only impacts future conversations.

TOPIC 2
2

👤 USER.md — who are YOU

USER.md is the user profile: name, role, preferences, communication style, languages, relevant contexts. He hasHard limit of 1,375 characters (~500 tokens). Exceeded, the agent stops accepting new entries until space becomes available.

✅ What makes sense here

  • •Name, role/profession, time zone, preferred language
  • •Style: short vs detailed answers, formal vs informal
  • •Stable technical preferences (languages, stack, operating systems used)
  • •Non-negotiable personal rules ("don't call me by my nickname", "avoid emojis in financial messages")

❌ What DOES NOT go into USER.md

  • •Current status ("I'm working on project X right now") — is volatile
  • •To-do list and ongoing tasks — I use another place (issues, planner)
  • •Passwords, tokens, API keys — go in~/.hermes/.env
  • •Agent Identity — this is SOUL.md
TOPIC 3
3

📓 MEMORY.md — personal notes from the agent

MEMORY.md is the agent's "working" memory. Notes on the environment, closed decisions, paths, shortcuts, facts learned. Hard limit of2,200 characters (~800 tokens). When it fills up, the agent itself consolidates redundant entries.

🧱 Good input examples

  • •"User's main repo is in~/projetos/site/, deploy via Hostinger."
  • •"User prefers commits in the formattipo: descrição curta, no extra prefixes."
  • •"Project X's Postgres Bank is accessible viadb.local.exampleport 5432."
  • •"Backup Cron runs at 3am, do not interfere at this time."

💡Tight limit is a feature

2,200 chars forces you to keep only what really matters. Perfect tool to avoid "messy memory that doesn't help at all". If you miss more space, you're probably putting the wrong thing there (status, task, history).

TOPIC 4
4

🔧 Tool memory: add, replace, remove

The agent manages its own memory viatoolmemory. Three operations available. You don't edit files manually all the time — you ask in natural language and the model triggers the right operation.

add

Adds new entry. Phrase:"Keep in mind that I prefer Postgres for new projects."

replace

Replaces existing entry. Phrase:"Update the entry about my time zone, now I'm in Lisbon."

remove

Deletes obsolete entry. Phrase:"Remove what you know about project X, I discontinued it."

🤖Self-consolidation when filling

When MEMORY.md or USER.md reaches the limit, the agent performs a consolidation pass: it joins redundant entries, rewrites more concisely, discards what has become obsolete. You can manually force:"Clean up MEMORY.md, add anything redundant."

TOPIC 5
5

🔍 session_search: SQLite + FTS5

Every conversation goes throughSQLite with full-text index (FTS5) em ~/.hermes/state.db. The toolsession_searchallows you to search for old conversations by keyword — it's the complementary "long memory".

📚 When to use each one

  • MEMORY.md:durable and relevant fact for MANY sessions. Always charged.
  • USER.md:your profile. Always charged.
  • SQLite + FTS5:specific conversation that may or may not be useful again. Searched on demand.

💡Ready sentence

"Search the session history for any conversation where we talked about <topic> and summarize the main points."The agent runssession_searchand brings you the consolidated material without inflating memory.

💾Backup matters

Backupstate.db= backup of the entire agent history. Include this file in the VPS backup routine. Lost, lost the past.

TOPIC 6
6

🚫 What to NEVER save in memory

Poorly used memory is not just waste — it becomesactive noisewhich harms the agent's reasoning throughout the session. And worse: secret saved there can leak in logs or in calls to remote models.

🚨 “No” list

  • ✗Secrets, tokens, API keys, passwords— go on~/.hermes/.env, never in memory files
  • ✗Temporary task— issue tracker, planner or current conversation
  • ✗Current status ("I'm on vacation")— it will be wrong in two weeks
  • ✗Conversation history— it's already in SQLite, don't duplicate it
  • ✗Long procedures— become a skill (full Markdown fits there)
  • ✗Agent personality— go to SOUL.md

🎯"Will it serve me in 6 months?" test

Before saving, ask: "will this be useful to me in 6 months, in any session, regardless of the task?" If yes → memory. If not → elsewhere.
