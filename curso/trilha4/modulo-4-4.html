⏰ Why security automation matters

Manual auditing is goodin the first two weeks. Then it became "I'll do it on the weekend", and then "after the holidays", and then never again. Cron solves it: the machine doesn’t forget, it doesn’t get tired, it doesn’t postpone.

⚖️ Manual vs automated

Manual

  • • Depends on your discipline
  • • Postpone in times of stress
  • • Fails when you need it most
  • • Inconsistent result

Automated

  • • Cron runs alone
  • • There is no bad mood or vacation
  • • Always runs on time
  • • You only review the report

⏱️Optimal frequency

  • •Heavy(full audit) — 1x per week
  • •Light(RAM/disk heartbeat) — every 5 min
  • •Quarterly(token rotation) — every 90 days
  • •Annual(full architectural review) — once a year

Very high frequency becomes ignored noise. Too low loses the event. Caliber.

📜 Skill audit-vps

Instead of describing the procedure every time, encapsulate it in aSKILL.md em ~/.hermes/skills/auditoria-vps/. Clear procedure = predictable execution. You review a consistent report.

📄Structure of SKILL.md

---
name: auditoria-vps
description: Auditoria semanal de segurança e saúde da VPS
version: 1.0.0
metadata:
  hermes:
    tags: [security, devops, audit]
    category: security
---
# Auditoria VPS

## When to Use
Cron semanal de domingo 9h, ou sob demanda em
incidente.

## Procedure
1. Portas abertas: `ss -tlnp` e `ufw status verbose`
2. Processos top: `ps aux --sort=-%mem | head -20`
3. Espaço em disco: `df -h` e `du -sh /var/log/*`
4. Logs estranhos: `tail -100 /var/log/auth.log`
5. fail2ban: `fail2ban-client status sshd`
6. Atualizações: `apt list --upgradable`
7. Containers: `docker ps -a`, `docker stats --no-stream`

## Verification
- Relatório tem todas as 7 seções
- Anomalias destacadas no topo
- Sugestões de ação com prioridade

## Pitfalls
- Logs de auth.log podem ser longos — pegar só ~100 linhas
- df -h em ZFS pode mostrar números enganosos
- Containers parados também precisam ser auditados

🎯 The 7 main checks

  1. 1.Open doors— any new ones that shouldn’t be there?
  2. 2.Suspicious processes— top CPU/RAM consumers
  3. 3.Disk space— partitions close to 90%
  4. 4.Strange logs— auth.log, syslog, kern.log
  5. 5.fail2ban status— how many IPs banned this week?
  6. 6.Outdated packages— security updates pending
  7. 7.Containers— any restarting, with OOM, stopped unexpectedly?

📅 Weekly Security Cron

Sunday 9am is the ideal slot: you wake up on the weekend with coffee, read the report, take action if necessary. No pressure of the work week. Hermes injects the skill into the fresh session and runs it alone.

⌨️Command

/cron add "every Sunday at 09:00" \
  "Rode auditoria-vps e me mande o relatório no Telegram. Destaque anomalias no topo." \
  --skill auditoria-vps \
  --deliver telegram

🚩 Important Flags

  • --skill auditoria-vps— injects the skill into the fresh cron session
  • --deliver telegram— deliver the report on Telegram (ordiscord, all, origin)
  • --workdir /path— sets cwd if there is specific context
  • --name "auditoria-semanal"— names the job for easy management

🔧 How it works internally

  1. 1.Scheduler reads~/.hermes/cron/jobs.jsonevery tick
  2. 2.See job due: Sunday 9am arrived
  3. 3.Start AIAgent fresh session
  4. 4.Inject the skillauditoria-vpsat system prompt
  5. 5.Run the prompt until completion
  6. 6.Delivery on Telegram
  7. 7.Updatenext_run_at

💓 Cron every 5 min on --no-agent

Mode--no-agentruns script without calling LLM.Zero cost. A Bash script checks RAM and disk, exits silently if OK, returns error if it passes the threshold. Very high signal-to-noise.

📜memory-watchdog.sh

#!/bin/bash
# Checa RAM e disco. Silent se OK, exit 1 se crítico.

RAM_USED=$(free | awk '/Mem:/ {print int($3/$2 * 100)}')
DISK_USED=$(df / | awk 'NR==2 {print int($5)}')

if [ "$RAM_USED" -gt 90 ]; then
  echo "ALERTA: RAM em ${RAM_USED}%"
  exit 1
fi

if [ "$DISK_USED" -gt 85 ]; then
  echo "ALERTA: Disco em ${DISK_USED}%"
  exit 1
fi

# Tudo OK, sai sem output (silent tick)
exit 0

📅 Cron command

hermes cron create "every 5m" \
  --no-agent \
  --script memory-watchdog.sh \
  --deliver telegram \
  --name "memory-watchdog"

Behavior:

  • •empty stdout + exit 0→ silent tick, nothing is delivered
  • •non-zero exit→ error alert delivered on Telegram
  • •Scripts.sh/.bashrotate under/bin/bash

💡Why --no-agent

Every 5 min × 30 days = ~8,640 runs/month. With LLM, this would burn significant budget. With--no-agent, zero cost. Use LLM only when you need reasoning.

🔄 Token rotation

Token leaked months ago and not exchanged is a silent vulnerability. Even if you never know about the leak, periodic rotation ensures that the exposure window is limited.90 daysis the reasonable standard.

⌨️Reminder cron

/cron add "every 90 days" \
  "Hora de rotar tokens. Liste:
   1. GitHub fine-grained tokens (verifique expiração)
   2. OpenRouter API keys
   3. OpenAI project keys
   4. Telegram bot tokens (se houve incidente)
   5. SSH keys (verificar authorized_keys)
   Avise quais foram trocados. Atualize MEMORY.md."

📋 Rotation procedure

  1. 1.Generate new key on the provider
  2. 2.Test the new key in an isolated environment
  3. 3.hermes config set KEY nova_chave
  4. 4.Restart gateway:hermes gateway restart
  5. 5.Confirm that it is working (send a test message)
  6. 6.Revoke the old key at the provider
  7. 7.Document the rotation inMEMORY.md

⚠️Don't skip step 6

Change the new key butdo not revokethe old one = having two valid keys circulating. Revoke is what effectively closes the exposure window.

TOPIC 6

📒 Post-incident becomes MEMORY.md

When something went wrong — token leaked, cron consumed infinite loop, destructive command passed without review — ask the agent:"Save this on MEMORY.md as a lesson: 'don't repeat this mistake'."Next session starts with remembering.

📝 Entry Template

## Incidente 2026-04-12
- Sintoma: cron de "summarize feeds" entrou em loop chamando GPT-4 a cada 100ms.
- Causa-raiz: skill blogwatcher mal escrita não tratou erro de RSS 503,
  re-tentou indefinidamente.
- Correção: adicionado retry com exponential backoff e limite de 3 tentativas.
- Lição: toda skill que faz HTTP precisa retry com cap. Hard cap de gasto
  no provedor é a última linha de defesa.

📊 Recommended structure

  • •Date— when it happened
  • •Symptom— what was observed
  • •Root cause— what actually caused
  • •Correction— what was done
  • •Lesson— generalized principle for future situations

📦Memory Limit

MEMORY.md has ~800 tokens (2,200 chars). When it fills up, the agent consolidates old entries to make room. General pattern persists even when specific incident comes out. Use reusable language: "every skill that does HTTP needs to retry with cap" is valid for years.

💡Why does this matter

It's the cheapest way to do postmortems. No need to call a meeting, write a formal document, create Jira."Saved as a lesson"and follow. Reduces recidivism. It becomes persistent learning between sessions. The agent remembers it next time, even without you having to remember.
