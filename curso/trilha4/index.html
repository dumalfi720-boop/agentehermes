🔐 Tokens, .env and hermes config set

Where to put credentials, which .env does Hermes read, how to prevent them from appearing in chat history, and how to keep logs without leaking secrets.

What is it:

Everything you paste into the Hermes chat goes to SQLite (sessions). Pasted token is indexed, searchable via session_search and copied in backups. Forever.

Why learn:

Carelessly leaked tokens are the most common attack vector. You pass the learning along, someone with access to SQLite (or the backup, or a log) pulls the key.

Key concepts:

Always usehermes config set KEY value. If you pasted by mistake, revoke the credential immediately and generate a new one. Don't try to "clear" the history — assume it was leaked.

What is it:

Command that writes the variable directly to/root/.hermes/.env(inside the container) without going through the chat history or logs.

Why learn:

It's the safe way. The CLI reads the value from stdin, persists it to .env, and commits without echoing. Works on laptop, VPS and Docker.

Key concepts:

Examples:hermes config set TELEGRAM_TOKEN xxx, hermes config set OPENROUTER_API_KEY xxx. When running in Docker, execute in the container before:docker exec -it hermes hermes config set ....

What is it:

In Docker, the .env from the project root on the host does not reach the process. Hermes reads the .env of/root/.hermes/.envinside the container (volume mounted in~/.hermeson the host).

Why learn:

It's the community's #1 confusion. You edit the "wrong" .env, restart, nothing changes, you think it's a Hermes bug — it's just the wrong path.

Key concepts:

Check withdocker exec -it hermes env | grep TELEGRAM. If the var does not appear, the volume is not mounting correctly or the name is wrong. Error logs:~/.hermes/logs/errors.log.

What is it:

To delete an old key, rename it or organize comments, edit directly:docker exec -it hermes nano /root/.hermes/.env.

Why learn:

O config setjust overwrite or add. It has no friendly "delete" subcommand. When the key is no longer used (you have switched providers), delete it manually to reduce the attack surface.

Key concepts:

Always backup before:cp /root/.hermes/.env /root/.hermes/.env.bak. Restart the gateway afterwards:hermes gateway restart.

What is it:

On GitHub you choose between classic token (broad access) and fine-grained (scope per repository, mandatory expiration). At OpenAI you define project key scopes.

Why learn:

Leaked token is a loss proportional to the scope. Classic full-access token exposes all your repos. Fine-grained in repo X only exposes X.

Key concepts:

Rule: start with minimum scope, expand when the agent complains about "permission denied". Always set expiration (90 days is a reasonable default).

What is it:

Hermes automatically redacts environment variable values ​​when they appear in error messages.~/.hermes/logs/errors.lograrely has the "real" token exposed.

Why learn:

You can ask for support, paste log chunks, post to the community Discord — without leaking credentials. But always check before sharing.

Key concepts:

Self-redaction covers known names (TELEGRAM_TOKEN, *_API_KEY). Hardcoded values ​​in code or free message may escape. Before posting log, open and read.

MODULE 4.2
4.2~30 min

🛡️ Firewall, ports and IP allowlist

Your VPS is subject to automated scanning all the time. Layered defense + ufw + SSH hardening reduces noise by 99% and closes ports that no one needs.

What is it:

Three layers: 1) provider firewall (Hostinger/AWS), 2) ufw on the host (Ubuntu), 3) Docker container isolation. Each blocks something that the other missed.

Why learn:

Trusting just one is fragile. If you turn off ufw temporarily to debug and forget about it, the provider's firewall still has you covered.

Key concepts:

Principle: nothing exposed without a clear reason. Every open door must have documented justification.

What is it:

Hostinger offers a firewall in its own panel (and Upguard as a monitoring service). Blocks traffic before reaching the VM, reducing load and log noise.

Why learn:

You don't need to wait for the package to arrive in Ubuntu to be discarded. Provider already filters. You save CPU, bandwidth and log lines.

Key concepts:

Recommended minimum: block everything, open only 22 (SSH), 80/443 (web) and specific ports you use. Review every 90 days.

What is it:

Uncomplicated Firewall — user-friendly iptables wrapper. In four lines you close the VPS to the world:ufw default deny, ufw allow 22, ufw allow 443, ufw enable.

Why learn:

It's the most granular control you have on the VM side. Closes traffic that arrived through the provider's firewall (e.g. another neighboring container).

Key concepts:

Alwaysufw allow 22beforeufw enable— under penalty of being locked out. Check withufw status verbose.

What is it:

Three adjustments in/etc/ssh/sshd_config: PasswordAuthentication no(key-only),Port 2222(non-default) and install fail2ban (ban IPs that try to brute force).

Why learn:

99% of attacks against VPS are automated SSH scans on 22 with a weak password. Taking 22 out of the equation rules out the majority. fail2ban solves the rest.

Key concepts:

Before disabling password, ensure that your key works. Before changing the port, open it in the provider's ufw and firewall. Test in a second session before closing the first.

What is it:

Restrict SSH to known IPs only. Hostinger allows via console, AWS via Security Groups, ufw viaufw allow from 1.2.3.4 to any port 22.

Why learn:

If you only access it from two places (home + office), there's no reason for the whole world to try SSH. Reduces attack surface to almost zero.

Key concepts:

Be careful with dynamic IP — if your house changes IP every morning, you'll be locked out. Solution: Fixed IP via VPN or Tailscale, and only releases the VPN IP.

What is it:

You ask in natural language: "Audit the firewall and propose hardening." Hermes rollsufw status, ss -tlnp, lists active ports, compares them with what makes sense and suggests changes.

Why learn:

You don't need to become an SRE. It lists, explains and suggests. You review and approve. Keeps the operation within its available time scope.

Key concepts:

Permission to run commands in the terminal must be configured. Always review before making critical changes (especially to SSH and firewall).

🪪 Principle of least privilege + separate accounts

Treat the agent like a new employee: you don't give full access on day 1. Minimum scopes, dedicated accounts, limited card, and agent-appointed keys.

What is it:

Onboarding mentality applied to software: day 1 with minimum scope, gradual expansion as reliability is proven. Just like a human intern — just with lower latency.

Why learn:

Reduces blast radius. Agent error in the first two weeks is limited loss. Error with full permission on the first day can break production.

Key concepts:

Day 1: read files, send Telegram. Week 2: commit to specific repos. Month 1: deployment in staging. Month 3: production with review.

What is it:

Create a new email just for the agent. AgentMail is a service created for this purpose; Dedicated Gmail also works. Never use your main account.

Why learn:

If the agent's account is compromised, the blast radius is only it. Your personal identity, bank recovery, 2FA of everything — stays intact.

Key concepts:

Also use for login to third-party services (OpenRouter, Agent's GitHub). Centralizes "agent's" credentials away from yours.

What is it:

In OpenRouter (and most providers), you create named keys.hermes-pessoal, hermes-marketing, hermes-financeiro— one per agent.

Why learn:

When something goes wrong, you know which agent burned tokens. You can revoke one key without affecting others. Spend audit becomes granular.

Key concepts:

Never share the same key between different Hermes. Even if it's lazy, it's the kind of shortcut that pays dearly later.

What is it:

Use a virtual card (Nubank, Revolut, Privacy.com) with a small limit just for agent subscriptions. Configure expense alert and monthly hard cap on the provider.

Why learn:

Infinite loop of a poorly written skill can burn hundreds of dollars in one night. Hard cap prevents it. Alert warns you before damage occurs.

Key concepts:

In OpenRouter: setmonthly limitper key. From OpenAI:usage limitsby project. On the virtual card: hard limit of R$200/month for experimental agents.

What is it:

When Hermes asks to run a command or use a sensitive tool, it offers three levels: approve once, approve the entire session, or approve forever.

Why learn:

Each choice has trade-offs.onceIt's safe but annoying (asks all the time).alwaysIt's convenient but becomes a permanent backdoor.sessionis the middle ground.

Key concepts:

Rule: destructive commands (rm -rf, git push --force) → once. Read commands (ls, cat) → always. Rest →session.

What is it:

Each Hermes (personal, marketing, finance) should have its own isolated credential block. Separate containers, separate .env, separate keys, separate profiles.

Why learn:

An agent's commitment does not cascade. Each one's audit is clean. You can retire an agent without changing the others.

Key concepts:

docker-compose pattern: one service per agent, with volume~/.hermes-pessoal, ~/.hermes-marketing, etc. Document who has access to what inSECURITY.md.

4.4~25 min

📋 Recurring audit + weekly skill

Security that depends on your failed memory. Cron + skill = audit that happens alone. Health heartbeat, token rotation and post-incident learning.

What is it:

Manual auditing is good for the first two weeks. Then it became "I'll do it on the weekend" and never again. Cron solves it: the machine doesn’t forget, it doesn’t get tired, it doesn’t postpone.

Why learn:

Security is a problem of habit. If it depends on your discipline, you fail. If it depends on the machine, it works. You just review the report.

Key concepts:

Ideal frequency: heavy once a week, light every 5 minutes. Too short an interval becomes ignored noise. Too long loses the event.

What is it:

Skill in~/.hermes/skills/auditoria-vps/SKILL.mdwhich checks: open doors (ss -tlnp), suspicious processes (top consumers), disk (df -h), strange logs (auth.log, syslog).

Why learn:

Clear procedure = predictable execution. You review a consistent report, you don't try to remember what to check out of nowhere.

Key concepts:

SKILL.md has YAML front matter (name, description, tags). Procedure lists the checks. Verification confirms that it ran. Pitfalls notes common false positives.

What is it:

Command:/cron add "every Sunday at 09:00" "Rode auditoria-vps e me mande o relatório no Telegram" --skill auditoria-vps.

Why learn:

Sunday 9am is the ideal slot: you wake up on the weekend with coffee, read the report, take action if necessary. No pressure of the work week.

Key concepts:

Use--skillto inject the skill into the fresh session.--workdirif there is folder specific context.--deliver telegramguarantees delivery.

What is it:

Mode--no-agentruns script without calling LLM. Zero cost. A Bash script checks RAM and disk, exits silently if OK, returns error if it passes the threshold.

Why learn:

Cheap watchdog. You find out before the site goes down. No alert = okay. Alert = immediate action. Very high signal-to-noise.

Key concepts:

Command:hermes cron create "every 5m" --no-agent --script memory-watchdog.sh --deliver telegram --name "memory-watchdog". Script.shwheel in/bin/bash.

What is it:

Every 90 days, generate new keys (GitHub, OpenRouter, Telegram bot if necessary) and revoke the old ones. Cron remembers; you perform.

Why learn:

Token leaked months ago and not exchanged is a silent vulnerability. Periodic rotation ensures that even an undetected leak has a limited window.

Key concepts:

Cron:/cron add "every 90 days" "Rote os tokens GitHub, OpenRouter e Telegram. Avise quais foram trocados.". Document each rotation in MEMORY.md.

What is it:

When something went wrong (token leaked, cron consumed infinite loop, destructive command approved without review), ask the agent: "Save this to MEMORY.md as a lesson".

Why learn:

It's the cheapest way to do postmortems. Next session starts with remembering. Reduces recidivism. It becomes persistent learning.

Key concepts:

Input pattern: date + symptom + root cause + correction action. Limit ~800 tokens — old entries are dropped when memory fills, but the pattern persists.
