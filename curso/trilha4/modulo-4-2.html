🧅 Layered defense

No layer alone is enough. The provider's firewall may have a misconfigured rule. ufw may be turned off by mistake. Container can run with--privileged. Three independent layersguarantee that an error in one is covered by the next.

🛡️ The 3 layers

1.
Provider firewall(Hostinger, AWS Security Group, Hetzner Cloud Firewall) — Blocks traffic before the VM. Cheaper in CPU and logs.
2.
ufw on Ubuntu host— Granular control within the VM. Covers traffic from neighboring containers and fine-grained rules per port.
3.
Container insulation— Hermes on Docker without--privileged, without useless mounts, with standard network bridge.

📐Rule of thumb

Every open door must have documented justification. If you can't explain in one sentence whyXis open, close.

TOPIC 2
2

🏰 Hostinger Upguard/built-in firewall

Hostinger offers a firewall directly from the VPS panel. Upguard adds scanning and alerts. Block the packagebeforeit reaches your VM — saves CPU, bandwidth and log lines in Ubuntu.

⚙️ Minimum recommended configuration

  • •Default:deny inbound
  • •Allow:22 (SSH) — ideally only from your IP
  • •Allow:80, 443 (web) if you expose dashboard or nginx
  • •Outbound:everything open (the agent needs to call APIs)

📅Maintenance

Review the rules every 90 days. Ports that you "opened to test" and forgot about are the most common incident vector on personal VPS.

TOPIC 3
3

🔥 ufw on Ubuntu

Uncomplicated Firewall— friendly iptables wrapper. In four lines you close the VPS to the world and open only what you need. It's the most granular control you have on the VM side.

⌨️Minimum configuration

sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow 22/tcp        # SSH
sudo ufw allow 443/tcp       # HTTPS (se for expor)
sudo ufw enable
sudo ufw status verbose

⚠️Beginner care

  • •Always ufw allow 22beforeufw enable— under penalty of locking yourself out of the VPS
  • •If you changed the SSH port to 2222, open 2222 before closing 22
  • •Have an SSH session open while testing — if you lock out, you can still roll back

💡Useful commands

sudo ufw status numbered    # listar regras com índice
sudo ufw delete 3           # apagar regra 3
sudo ufw allow from 1.2.3.4 to any port 22  # só seu IP
sudo ufw disable            # desligar (em emergência)
TOPIC 4
4

🔑 SSH hardening

99% of attacks against VPS are automated SSH scans on port 22 with a weak password. Three simple adjustments eliminate almost all of this.

🛠️ The 3 adjustments

  1. 1.
    Key-only:edit/etc/ssh/sshd_config, PasswordAuthentication no. Ensures your key works first.
  2. 2.
    Non-standard port: Port 2222(or other). It's not real security, but it drastically reduces noise in the logs.
  3. 3.
    fail2ban: install withapt install fail2ban. Ban IPs after N failed attempts, automatically.

📋Step-by-step

# 1. Garantir chave funcionando (de outra sessão)
ssh-copy-id usuario@vps

# 2. Editar config
sudo nano /etc/ssh/sshd_config
# - PasswordAuthentication no
# - Port 2222
# - PermitRootLogin no

# 3. Abrir nova porta no firewall
sudo ufw allow 2222/tcp

# 4. Restart SSH (mantenha sessão atual aberta)
sudo systemctl restart sshd

# 5. Testar em nova sessão antes de fechar antiga
ssh -p 2222 usuario@vps

# 6. Fechar a 22 quando confirmou
sudo ufw delete allow 22/tcp

# 7. fail2ban
sudo apt install fail2ban
sudo systemctl enable --now fail2ban

⚠️Order matters

Neverclose the 22 before testing the 2222 working.Neverdisable password before confirming that your key is authorized. Always test on a second session before killing the first.

TOPIC 5
5

📍 IP allowlist when possible

If you only access the VPS from two places (home + office), there is no reason for the whole world to try SSH. Restricting SSH to known IPs reduces the attack surface to almost zero.

🌐 How to apply

  • Hostinger:panel → Firewall → add rule "allow 22 from 1.2.3.4"
  • AWS:Security Group inbound rule, source = "My IP"
  • ufw: sudo ufw allow from 1.2.3.4 to any port 22
  • Hetzner:Cloud Firewall, source specific IPs

⚠️Beware of dynamic IP

If your house changes IP every morning (common residential provider), you will lock yourself out of the VPS the next morning. Solutions:

  • •Tailscale— Mesh VPN, virtual fixed IP
  • •Commercial VPN with static IP
  • •Bastion/jump hostwith fixed IP
  • •Dynamic DNS+ script that updates ufw automatically

💡Ideal combination

Tailscale + ufw allow only from range 100.x.x.x (Tailscale CGNAT). SSH only through Tailscale. Rest of the world doesn't even see port 22 open.

You don't need to become an SRE. Hermes can run the audit, list the current state and suggest hardening. You review and approve change by change.

💬 Suggested prompt

"Audit the firewall and SSH configuration of this VPS. List open ports (ss -tlnp, ufw status), services listening, strange rules. Compare with what makes sense for a VPS running only Hermes Agent. Propose hardening in order of priority. Do not make changes without my approval."

📊 What he usually thinks

  • •Open port 8080 from some old experiment
  • •SSH still accepting password
  • •Docker container exposed directly in 0.0.0.0
  • •fail2ban not installed or jail disabled
  • •Ubuntu packages not updated in weeks

⚠️Always review first

Changes to SSH and firewall can lock you out of the VPS. Permission for the agent to execute sensitive commands must beonce ou session, neveralways. Always keep an SSH session open while approving changes.
