MODULE 4.3

🪪 Principle of least privilege + separate accounts

Treat the agent as a new employee: day 1 with minimum scopes. Dedicated email, named keys, limited card, granular permissions and isolation between different Hermes.

TOPIC 1
1

👷 Treat like a new employee

The mentality that unlocks everything: Hermes is not an "infallible robot", it isnewly hired employee with low latency and zero malice, but also zero context. You don't give an intern full access on day 1. Same logic.

📈 Gradual trust (timeline example)

Day 1:
Read files in the repo, send messages on Telegram, consult public APIs. No writing on anything.
Week 2:
Commit to branch, open PR, write to repo files. Never push directly to main.
Month 1:
Deploy in a staging environment. Read-only database access. Low risk cron.
Month 3:
Production under review. Read-write database on specific tables. Crons that affect customers.

🎯Why it works

Agent error in the first few weeks is limited loss. Error with full permission on the first day can break production, corrupt database, leak data. Trust gradually expanded is trust gained.

TOPIC 2
2

📧 Separate accounts

Create new identity for the agent.Neveruse your main account (personal Gmail, main GitHub). If the agent is compromised, you want minimum blast radius.

📬 Dedicated email

  • •AgentMail— service created specifically for agents
  • •Dedicated Gmail— e.g.:seunome.hermes@gmail.com
  • •ProtonMail— if you want more privacy

🐙 Agent GitHub

  • •Separate GitHub account with agent email
  • •Added as a collaborator in specific repos
  • •Never owner of the important repos

⚠️Why separate

If an agent's account is compromised, their personal identity, bank recovery, 2FA everything — stays intact. The attacker only takes what the agent had access to. You revoke, exchange keys and move on.

TOPIC 3
3

🏷️ Agent-appointed keys

OpenRouter, OpenAI, Anthropic, GitHub — they all allow you to name keys. Use this. One key per agent, with a descriptive name. When something goes wrong, you know who burned tokens, who was compromised, which one to revoke.

🏷️ Naming standard

  • hermes-pessoal— everyday agent, low risk
  • hermes-marketing— social media automation, content
  • hermes-financeiro— reports, alerts, separate accounts
  • hermes-cliente-acme— exclusive agent of a client
  • hermes-experimental— tests, new skills, low limit

📊Benefits of isolation

  • •Granular spend audit— you see exactly which agent burned X dollars
  • •Surgical revocation— erases one key without affecting the others
  • •Independent limits— experimental agent has a cap of $10/month, staff has a cap of $50
  • •Separate logs— provider analytics shows patterns for each agent

⚠️Do not share

"I'm going to use the same key on both agents to save time" → no. It's exactly the shortcut that pays dearly later. One leak contaminates the others, the audit gets messed up, the revocation brings everything down.

TOPIC 4
4

💳 Limited credit card

Infinite loop of a poorly written skill can burn hundreds of dollars in one night. Hard cap prevents damage. Alert warns you beforehand.Always use a virtual card with a low limitfor pay-per-token services.

💳 Virtual card

  • •Nubank Ultraviolet— virtual with custom limit
  • •Revolut— disposable cards
  • •Privacy.com(US) — disposable cards with monthly cap
  • •BTG More— virtual with adjustable limit

🚦 Hard cap on providers

  • •OpenRouter:monthly limit per key
  • •OpenAI:usage limits per project
  • •Anthropic:spending threshold per workspace
  • •Hostinger:VPS auto-shutdown when budget is reached

📊 Strategy on 3 levels

  1. 1.
    Alertat 50% of the budget — you are warned
  2. 2.
    Soft limitat 80% — agent is warned, still works
  3. 3.
    Hard capat 100% — key to accept requests

⚠️Real story

Poorly written skill went into a loop calling GPT-4 every 100ms. Without a hard cap, he burned $800 in 4 hours overnight. With a hard cap at $50, it would have stopped in 15 minutes with a loss of $50. Use hardcaps.

TOPIC 5
5

✋ Granular permissions: once / session / always

When Hermes asks to run a command or use a sensitive tool, it offers three levels of approval. Each has trade-off between security and friction. Knowing which one to use when is half the operational discipline.

🔒 Once

Approves only this specific execution. Next time ask again.

Use when:destructive commands (rm -rf, git push --force), changes in production, irreversible actions.

⏱️ Session (current session)

Approve until the session ends. When you restart it asks again.

Use when:work focused on a task, commands from the same family repeated, package installations during setup.

♾️Always

Permanently approves. It becomes a backdoor without any further questions.Careful.

Use when:pure read commands (ls, cat, git status), tools that you trust 100% (sending Telegram to yourself).

🎯 Rule of thumb

  • Reading(cat, ls, git status, curl GET) →always
  • Controlled writing(git commit, create file) →session
  • Destructive(rm, push --force, drop table) →once
  • External at cost(create VM, buy API credits) →once
TOPIC 6
6

🔀 Multi-agent = multiple keys

Each Hermes (personal, marketing, financial) must besilo. Separate container, separate volume, separate .env, separate keys, separate Telegram/Discord profile. One's commitment doesn't cascade.

📁Disk layout

/home/user/
├── hermes-pessoal/
│   ├── docker-compose.yml
│   └── data/.hermes/      → /root/.hermes do container
├── hermes-marketing/
│   ├── docker-compose.yml
│   └── data/.hermes/
└── hermes-financeiro/
    ├── docker-compose.yml
    └── data/.hermes/

🧪 docker-compose by agent

services:
  hermes-pessoal:
    image: nousresearch/hermes-agent:latest
    container_name: hermes-pessoal
    volumes:
      - ./data/.hermes:/root/.hermes
    restart: unless-stopped

📋 Isolation checklist

  • ✓Different containers (container_namedifferent)
  • ✓Different volumes (each one points to its own~/.hermes-X)
  • ✓Different API keys (one per agent)
  • ✓Different Telegram bots (one per agent)
  • ✓Different Email/GitHub (when it makes sense)
  • ✓Document everything inSECURITY.mdin central repo

💡Document

Create oneSECURITY.mdin the central repo with: agent name, purpose, keys used, scope of each key, responsible human owner, date of last rotation. In 6 months you won't remember.
