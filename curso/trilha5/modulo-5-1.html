🆚 Hermes vs Claude Code

Five concrete differences between Hermes and Claude Code, with scenarios where each side wins. It's not "one or the other" — it's understanding the niche of each product to build the right stack.

📱 Native Messaging Gateway

Hermes hasnative gatewaywhich connects the agent to 20+ messaging platforms — Telegram, Discord, WhatsApp, Slack, Signal, iMessage (BlueBubbles), Microsoft Teams, Matrix, Google Chat and others. Claude Code operatesexclusively in the terminal: no gateway, no bots, no message delivery outside the terminal.

🌐What does this change in practice

  • Hermes:you walk down the street, send audio on Telegram, it transcribes, reasons, executes skills, responds.
  • Claude Code:It only works when you sit at the PC and open the terminal — it doesn't respond on the cell phone alone.
  • Claude Code equivalent:would require an extra stack (custom bot + bridge + remote execution), assembled by hand.

💡Command that illustrates

hermes gateway setupruns an interactive wizard that asks which platform, requests token, generates config, tests. In 5 minutes the agent is responding on Telegram. In Claude Code there is no equivalent — what you have is a CLI instance running locally.

⏰ Native Cron

Hermes includesbuilt-in task schedulerwith direct delivery to the message channel. You say "every day at 8 am, send a summary of priorities on Telegram" and cron runs it. Claude Codedoesn't have cron— you would have to orchestrate via the system crontab, write wrapper script and manage delivery manually.

✓ Hermes (cron + gateway)

  • ✓"Every day at 8 am, summarize priorities on Telegram"
  • ✓"Friday 6pm, send a summary of the week on Slack"
  • ✓"Sunday, audit server and alert if anything suspicious"
  • ✓LLM reasons the output at runtime

✗ Claude Code (without cron)

  • ✗System crontab calls wrapper script
  • ✗You code delivery (curl for Telegram bot)
  • ✗Token management and manual context
  • ✗Operational grows a lot in scale
TOPIC 3
3

📚 Persistent Skills vs free CLAUDE.md

Hermes maintains skills likestructured YAML documents em ~/.hermes/skills/with progressive disclosure — only uploads what is important for the current task. Claude Code usesCLAUDE.mdand free system prompts, without indexed catalog or progressive disclosure.

🧬 Technical comparison

Hermes SKILL.md:YAML front matter (name, description, when_to_use), structured procedure, versioned in~/.hermes/skills/, synchronizable via Skills Hub.
Claude Code CLAUDE.md:single file in the project root, free system prompt, no catalog, read in full every session.
Trade-off:CLAUDE.md is simpler for small projects; SKILL.md scales better to hundreds of procedures without overflowing context.

💡Synergy

You can have CLAUDE.md (for Claude Code) and .hermes.md/skills/ (for Hermes) in the same repo. Each agent reads its format. No conflict.

TOPIC 4
4

🔌 Multi-inference provider

Hermes connects toOpenAI, Anthropic, OpenRouter, local models (Ollama/llama.cpp), MiniMax OAuth and Codex OAuth. Claude Code iscoupled to the Claude modelvia Anthropic API or Claude Pro/Max.

💰 Why it matters for cost control

  • Codex via ChatGPT Plus ($20/month):heavy use of non-pay-per-token coding. Hermes consumes this subscription.
  • OpenRouter:200+ models via single API, model exchange at runtime depending on the task.
  • Local model:Qwen, Llama, DeepSeek running on personal GPU — extreme privacy, zero inference cost.
  • Fallback:if Anthropic goes down, Hermes stays with OpenAI. Claude Code for.

⚠️Lock-in is a risk

Coupling production to a single provider is an operational risk. If the API goes down, the price goes up or the model is discontinued, you are held hostage. Multi-provider is continuity insurance.

TOPIC 5
5

📋 Multi-agent Kanban (v0.13.0)

v0.13.0 "The Tenacity Release" introducesmulti-agent Kanban orchestrationwith heartbeat, reclaim and zombie detection. Workers pick up tasks, mark statuses, return them to the pool if they get stuck. Claude Codedoes not have native multi-agent coordination.

🏗️ How Kanban works

Board:To Do/Doing/Done columns. Tasks are entered via To Do.
workers:agents take tasks from the top of To Do, move them to Doing, and execute them.
Heartbeat:worker proves it is alive every N seconds. If it stops, task is marked zombie.
Claim:another worker takes the zombie task, resumes from the saved state.
Done:completion moves to the Done column. You review the result.

💡When it matters

Multi-repo refactor (1 worker per repo). Process 1000 emails (parallel workers). Audit 50 servers (1 per host). Without Kanban, it becomes a compromise with an external queue (Redis, RabbitMQ).

🎯 Conclusion: scenarios where everyone wins

The question is not "which one to choose". It's "in which scenario each is better" — and the serious professional uses both.

🪽 Hermes wins when...

  • ✓You are on the move and want an agent via Telegram/WhatsApp
  • ✓Need scheduled automations (cron)
  • ✓Do you want an agent always on, 24/7, on VPS
  • ✓Need multi-provider (cost, fallback, location)
  • ✓Task calls for coordinated multi-agent Kanban
  • ✓Non-technical staff need to interact with agent

🤖 Claude Code wins when...

  • ✓You are sitting at the PC, inside the project
  • ✓Task requires deep code reasoning
  • ✓Complex refactor, PR review, intricate debugging
  • ✓Want Claude Sonnet 4.5/Opus model in IDE
  • ✓Workflow is interactive, not scheduled
  • ✓Single dev sitting doing deep work
