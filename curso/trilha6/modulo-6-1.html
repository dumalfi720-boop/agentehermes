🚫 Don't create too soon

The most common mistake made by those who start with Hermes is to create3-4 agents in the first monththinking he's "getting ready to climb." Result: no agent is well trained, everyone has shallow memories, and in 60 days 2 of them are abandoned.

📏"1st agent" heuristic

Just create onesecond agentwhen the first one has:

  • ✓At least5 skillsin regular use
  • ✓At least3 cronsdelivering value
  • ✓At least1 monthreal use (not just testing)
  • ✓At least1 timewhere you updated memory or soul to correct behavior

💡Why does this matter

Operate 1 solid agentteaches you the complete cycle: create skill, update memory, adjust soul, schedule cron, debug, rotate. These repeated cycles turn into muscle. Trying to learn the cycle with 3 agents at the same time dilutes learning and generates frustration.

TOPIC 2
2

🎯 Criteria for a new agent

When the criterion appears, separate without mercy. They exist4 concrete signsWhat's the point of creating a second agent instead of piling more things onto the first one?

🔑 Different keys

The new function requires a different account: another Telegram bot, another Gmail, another bank API key. Mixing violates the principle of least privilege.

🧠 Memory that cannot mix

Data from customer A leaking to customer B is a catastrophe. Personal memory leaking to the company agent, ditto. Isolate via container.

🎭 Very distinct function/tone

"Informal and debauched" soul for you doesn't match "professional and empathetic" for support. Even soul doesn't do both.

⚠️ Different risk scope

An agent who can pay bills is a different risk than an agent who only reads email. Set aside to reduce blast radius if something is compromised.

📋 Rule of thumb

If 2+ criteria are present, separate. If only 1, consider it — perhaps a new skill with limited scope will solve it without fragmenting it.

TOPIC 3
3

🗂️ Pattern by vertical

Recipe tested by dozens of operators. You don't need to invent the division - these are thecutouts that workin personal and small business operations.

🧘 Staff — assistant of the day

Summaries, reminders, agenda, assisted reading. Informal soul. Memory about yourself, personal projects, preferences. Access to your main Telegram.

📣 Marketing — brand tone

Soul = voice of the brand, with examples. Memory = content history, editorial calendar. Skills: respond to comments, suggest threads, review copy.

💰 Financial — restricted access

Separate bank keys. Private Telegram channel. Seriously soul. Skills: weekly report, due date alert, cost anomaly.

🎧 Support — FAQ + escalation

Empathetic and professional soul. Memory = Live FAQ. Skills: triage, response suggestion, escalation to human in critical cases.

⚙️ Oops — audit + backup

Root access to the VPS. Crons: weekly audit, daily backup, disk/memory monitor. Direct and technical soul.

✍️ Content — newsletter, posts

Soul = your authorial voice. Memory = examples of your good texts. Skills: review text, suggest variations, compile weekly newsletter.

TOPIC 4
4

🏢 Anatomy of the "building"

Simple and useful mental model: think aboutthree physical layers. Each one has clear responsibility, and confusing them is a source of recurring bugs.

A

VPS = the building

Physical resource (CPU, RAM, disk), public IP, bandwidth. Ubuntu LTS + Docker + nginx + ssh. You are the building manager — you take care of the entire building.

B

Docker container = the room

Isolation. Each agent has its container, its .env, its volumes. Restarting one room does not affect the others. Updating a version of Hermes is changing the contents of the room.

C

Agent Hermes = the employee

The person working inside the room. It has skills (procedures), memory (facts), soul (way of being). When you "update the agent" you are editing these Markdown files.

⚠️Classic confusion

"I put the key in the .env and it doesn't work." → You edited the .env of the building (host) when Hermes reads the .env of the room (container). Always confirm where you are editing:docker exec -it hermes env | grep TELEGRAM.

TOPIC 5
5

🚚 Migration between agents

Good news: Hermes keeps almost everything inMarkdown files. Skills, MEMORY.md, SOUL.md, USER.md — all readable, versionable, and portable. Migrating knowledge between agents becomes agit cloneand a tone adjustment.

📋 Migration Recipe

  1. Create a private GitHub repo:hermes-shared
  2. Inside:/skills, /templatesof soul/memory by vertical
  3. Each container doesgit cloneand only link what you use
  4. Skill shared by 5 agents: edited 1 time, everyone gets it in the next pull
  5. Vertical-specific skill: folder/marketing, /financeiro, etc.

💡Versioning tip

Treat skills as code: PR review before merging, changelog when behavior changes, simple semver (v1.0, v1.1). Prevents "new skill broke backup cron" without warning.

🪐 Multi-agent Kanban (v0.13.0)

The big news ofv0.13.0 "The Tenacity Release"is the multi-agent Kanban system. Instead of you orchestrating manually, an orchestrator profile creates a durable board, defines task cards, and multiple Hermes workers pick it up, work it, and close it.

📜 Quote from release notes

"Spin up a durable board, drop tasks on it, and let multiple Hermes workers pick them up, hand off, and close them out. Heartbeats, reclaim, zombie detection, retry budgets, and a hallucination gate keep the team honest."

💓 Heartbeats

Each worker pings the board. No heartbeat for X minutes = considered stuck. Task returns to the top.

🔄Reclaim

Task abandoned by a stuck worker is "claimed" by another available one. Job doesn't crash because of a zombie worker.

🧟 Zombie detection

Hanging process (0% CPU, no heartbeat) is detected and killed. Avoid ghost workers consuming resources.

🎯 Hallucination gate

Before closing card, validate that the output makes sense (tests pass, non-empty output, etc). Block "completed" without having done it.

🎯 Typical use cases

  • •Bulk content generation:1 orchestrator + 3 writing workers + 1 reviewer
  • •Data Analysis:divide dataset into N pieces, each worker processes one
  • •Code refactor:orchestrator describes change, workers apply per file
  • •Parallel search:N themes, N workers, consolidation at the end
