MODULE 6.1
6.1~30 min

🌳 When to create new agents

Don't start with 5 agents. Start with 1 and separate when criteria warrant. Decision tree, patterns by vertical and the path to multi-agent Kanban from v0.13.0.

What is it:

Start with 1 single agent, without multiplying until there is a concrete reason. Most people who create 3-4 agents in the first month end up abandoning 2 of them.

Why learn:

A single agent, well trained and used, is worth more than five empty ones. Operating 1 solid agent teaches you the cycle of skills, memory and crons before splintering.

Key concepts:

Heuristic: only create a second agent when the first one already has at least 5 regularly used skills, 3 useful crons and 1 month of real use.

What is it:

Four concrete criteria: (1) different API keys; (2) memory that cannot mix; (3) very distinct function/tone; (4) different risk scope.

Why learn:

Without criteria, they saw 5 agents doing almost the same thing. With discretion, each agent has clear responsibility and independent updating.

Key concepts:

Personal and financial mix tone (informal vs serious), keys (personal Telegram vs restricted channel) and risk (access to bank account). Justifies separation.

What is it:

Tested recipe: personnel (assistant of the day), marketing (brand tone), financial (restricted access), support (FAQ + escalation), ops (audit + backup), content (newsletter, posts).

Why learn:

You don't need to invent division. This is the cut that supports small and medium-sized operations. You can start with 2 verticals and add as you use them.

Key concepts:

Each vertical = own Docker container + own .env + own skills. Soul.md changes the tone; memory.md changes the facts; user.md changes who owns it.

What is it:

Mental model: VPS is the building (physical resource, IP, bandwidth), Docker container is the room (isolation, own .env), agent Hermes is the employee inside that room (skills, memory, soul).

Why learn:

This model avoids the classic confusion: "why doesn't my Telegram key work?" usually it's because you edited the building when you should have edited the room.

Key concepts:

Multiple containers on a single VPS is the standard. Each person exposes (or not) their own door. VPS backup protects all containers at once.

What is it:

Skills, MEMORY.md, SOUL.md, USER.md — all Markdown versioned. Migrating knowledge between agents means copying files, adjusting the tone in the soul and uploading a new container.

Why learn:

It makes multiplication cheap. You create the "marketing" agent by reusing 70% of the "staff", only changing the soul and some specific skills.

Key concepts:

Maintain a GitHub repository with shared skills. Each container doesgit cloneand only use the subdirectories you need.

What is it:

New feature in v0.13.0 ("The Tenacity Release"): an orchestrator profile creates a durable Kanban board, defines tasks, and multiple Hermes workers pick up, handoff, and close.

Why learn:

It is the path of real parallelism. You describe a large job, it is broken down into cards, and your agents execute it in parallel. Useful for research, content generation, data analysis.

Key concepts:

Heartbeats keep workers honest. Reclaim returns a stuck worker task. Zombie detection kills hanging processes. Hallucination gate prevents suspicious output commit.

MODULE 6.2

📊 Dashboard, gateway, tunnel, Kanban

Hermes has a complete web UI. Learn what fits in it, how to open via SSH or nginx, and when pure Telegram is still the best way.

What is it:

The web UI includes session view, list of active channels, crons (cron view), skill manager, plugins, configurations and — in v0.13.0 — multi-agent Kanban view.

Why learn:

Some tasks are easier visually: review last 50 sessions, activate/deactivate plugins, edit config files. Telegram is great for everyday life, but the dashboard is better for maintenance.

Key concepts:

Dashboard is just read/edit internal state. Everything it shows is also in archives.~/.hermes/.

What is it:

The dashboard goes up by default atlocalhost:8080inside the VPS. To access from your laptop:ssh -L 8080:localhost:8080 user@vpsand openhttp://localhost:8080.

Why learn:

SSH tunnel avoids exposing the dashboard publicly. Nginx + Cloudflare Tunnel is the alternative when you want fixed URL with authentication.

Key concepts:

Never open the dashboard door directly on the internet without authentication. Admin token is not a firewall. SSH key + tunnel is the secure default.

What is it:

Skill bundled "open-dashboard": you ask on Telegram"open my dashboard"and it runs the local SSH tunnel, waits for the port to open and gives you the linkhttp://localhost:8080.

Why learn:

Reduces friction. Instead of memorizing an SSH command, opening a terminal and typing — you speak. Maintains the agent as the single operating port.

Key concepts:

The skill assumes that your SSH key and host are configured locally. It doesn't work if the agent is on the VPS — only running locally on your Mac/Linux.

What is it:

Board visual (To Do, In Progress, Done) that shows task cards and which worker took each one. Added in v0.13.0 along with Multi-Agent Kanban.

Why learn:

When you run 2-3 workers in parallel, it becomes difficult to follow on Telegram. The Kanban view gives an immediate overview: what’s missing, what’s stuck, which worker is zombied.

Key concepts:

Cards show workers' heartbeat, retries count and state of the hallucination gate. Task marked zombie automatically returns to the top of the board.

What is it:

The plugins tab lists all installed plugins, version, status (active/deactivated) and buttons to update, deactivate or remove. Includes official (LCM, kanban) and community plugins.

Why learn:

Visual maintenance avoids making incorrect commands in the terminal. Updates are one click instead of manual pull + restart.

Key concepts:

Plugin updates can break skills that depended on the old version. Always look at the changelog before updating in production.

What is it:

Most of the daily use can be done via Telegram: requesting tasks, checking crons, reading summaries, adjusting memory. Dashboard is for maintenance, not operation.

Why learn:

Anyone who gets used to opening a dashboard for anything loses the practicality of the agent “in their pocket”. Telegram is the single point of contact.

Key concepts:

Rule: if you can order in text, order via Telegram. Only go to the dashboard when you need to edit long files, view graphs or sort dozens of Kanban cards.

🛠️ Complete troubleshooting

The 6 problems that appear in almost every Hermes installation — and the solution path for each one. Telegram, Docker, context, timezone and diagnosis via container terminal.

What is it:

Symptom: you send a message and receive nothing. Sequence:hermes gateway status→ read~/.hermes/logs/gateway.log→ checkTELEGRAM_TOKEN e TELEGRAM_HOME_CHANNELin the .env.

Why learn:

It is the most common problem. In 90% of cases it is the wrong token, home channel not configured or gateway stopped. Knowing the sequence saves 30 minutes each time.

Key concepts:

In Docker, also confirm that the container has access to the internet (firewall/iptables).hermes gateway setupre-run the wizard if something became inconsistent.

What is it:

Symptom:localhost:8080does not load on the laptop. On VPS without direct public IP, it is expected — you need to create the tunnel:ssh -L 8080:localhost:8080 user@vps.

Why learn:

A lot of people think it's bugged when in fact the only thing missing is the tunnel. Confirming that the Hermes process is running on the correct port avoids lost debugs.

Key concepts:

Alternative: nginx as reverse proxy with basic auth + HTTPS via Let's Encrypt. Good when multiple people need access.

What is it:

Classic Docker problem: the right .env stays in/root/.hermes/.envinside the container. On the host, mount:~/.hermes:/root/.hermes.

Why learn:

Anyone who places keys in the .env of the project root without a mounted volume sees the agent go up without reading anything and keeps hunting for bugs where there are none.

Key concepts:

Check withdocker exec -it hermes env | grep TELEGRAM. Self-redacting logs are in~/.hermes/logs/errors.log.

What is it:

Hermes automatically compresses long conversations before the model limit. Command/compressstrength./usageshows tokens. v0.13.0 shows compression count in status bar.

Why learn:

Symptom: agent "forgetting" things in the middle of the session. It is usually very aggressive or poorly configured compression. Switching to the LCM plugin eliminates loss.

Key concepts:

Use a cheap model for compression (google/gemini-2.5-flashvia OpenRouter). LCM plugin offers lossless compression without extra LLM call.

What is it:

Docker Container runs in UTC by default. Hermes needstimezone: "America/Sao_Paulo"in config.yaml orTZ=America/Sao_Pauloin the env vars.

Why learn:

Without timezone, cron for "09:00" triggers at 09:00 UTC = 06:00 in Brasília. You wake up to a message from the agent 3 hours earlier than expected and you don't understand why.

Key concepts:

Affects: timestamps in logs, cron scheduling, and the time injected into the system prompt. Use standard IANA string (America/Sao_Paulo, Europe/Lisbon, etc).

What is it:

Inside the container:docker exec -it hermes bashand use the CLIhermeswith subcommands:gateway status, cron list, config get, doctor.

Why learn:

When Telegram is unresponsive, you need a diagnostic port outside of it. Container terminal is the "safe way" to inspect everything.

Key concepts:

Logs with self-written secrets are in~/.hermes/logs/. You can send it to support without fear of a token leak.

MODULE 6.4
6.4~25 min

🔁 Continuous maintenance routine

The difference between those who use Hermes for 1 month and those who use it for 1 year: simple routine, honest updating of memory, and the philosophy that an agent is a colleague who learns — not a tool that ends.

What is it:

Three rules: (1) made a mistake 2x → updates skill or memory; (2) repeated instruction → becomes skill; (3) wrong tone → adjusts soul. Apply immediately, do not accumulate for later.

Why learn:

Without a rule, maintenance becomes a “when there is time” task — and it never is. With a simple rule, it becomes a reflex: "this became a problem twice, now save it".

Key concepts:

Magic phrase:"This has become a skill, save."The agent takes care of the rest. You don't need to remember the syntax of SKILL.md.

What is it:

Monthly, order:"read your memory", "read your soul", "list active skills", "list active crons". The agent himself presents a summary and suggests pruning.

Why learn:

It's a cheap audit. You see in 5 minutes what is outdated, what is duplicated and what has been orphaned. Without this, memory becomes a silent dump.

Key concepts:

Place a monthly cron:"suggest memory pruning and list 3 skills without recent use". You just approve or reject it.

What is it:

Memory.md accumulates old facts: old VPS, dead project, preference that has changed. The agent continues to act according to these facts and it seems "strange" or "wrong".

Why learn:

Before assuming code bug, open memory.md. 70% of Hermes' "strange behavior" is an outdated fact inside.

Key concepts:

Memory should be edited as a journal, not a log. When something changes in your life, ask:"updates memory: now I use X instead of Y".

What is it:

Debug protocol: 1) read memory.md, 2) read soul.md, 3) read skill involved, 4) only then check logs and config. Errors are often old instructions, not bugs.

Why learn:

You save hours of code debugging when the problem is text. Hermes is 90% Markdown — starting with text covers most cases.

Key concepts:

Phrase to use:"strange behavior in X. Audits memory, soul and related skill before I change the code."

What is it:

Cadence: quarterly token rotation; monthly skills review; weekly security audit (cron skill "security audit"). Everything scheduled, not manual.

Why learn:

Without cadence, nobody does it. With cron, it becomes automatic: the agent remembers you. It's the difference between "I intend to" and "it is happening."

Key concepts:

Expired tokens in .env are a frequent cause of a stopped gateway. Weekly audit catches open ports, new processes and suspicious skills.

What is it:

Hermes is not an app that you install and that's it. He's a colleague: he needs onboarding, context, correction and growth. Those who treat it as a static tool never take advantage of it.

Why learn:

The difference between 1 month and 1 year of Hermes is exactly this. Whoever keeps the relationship alive has an increasingly useful agent. Whoever treats it as a tool abandons it.

Key concepts:

Final mindset: you are the tech lead, he is the senior intern. You approve, correct and teach. He executes, remembers and suggests. Every month, more alignment.
