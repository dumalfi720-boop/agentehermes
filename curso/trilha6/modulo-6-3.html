📵 Telegram does not respond

Symptom: you send a message and receive nothing.Most common problem #1. In 90% of cases it is one of four causes. Let's go in the right order.

📋 Diagnostic sequence

  1. Is Gateway running?
    hermes gateway statusinside the container.
  2. Re-run wizard if inconsistent:
    hermes gateway setup
  3. Read logs:
    tail -f ~/.hermes/logs/gateway.log
  4. Check token and home channel:
    grep TELEGRAM ~/.hermes/.env

🔑 TELEGRAM_TOKEN

It comes from BotFather. If you revoked and created a new one, update it here. No valid token = gateway does not even start.

📮 TELEGRAM_HOME_CHANNEL

Chat ID where Hermes delivers cron results. Without this configured, automated messages become silent.

🐳 In Docker

Confirm that the container has access to the internet (firewall/iptables). Test:docker exec hermes curl -s https://api.telegram.org. If it doesn't return 404 (expected without path), there is a network problem in the container.

TOPIC 2
2

🌐 Dashboard does not open

Symptom:localhost:8080does not load. On VPS without direct public IP, it isexpected— you need to create the bridge. It's not a bug.

📋 Checklist

  1. Confirm process running on the correct port (within the VPS):
    ss -tlnp | grep 8080
  2. Create SSH tunnel from laptop:
    ssh -L 8080:localhost:8080 user@vps
  3. Accesshttp://localhost:8080in the browser
  4. If you want a fixed URL: configure nginx as a reverse proxy with basic auth + Let's Encrypt

🔄 Alternatives

  • •nginx + Let's Encrypt:Public fixed URL with HTTPS and auth
  • •Cloudflare Tunnel:without exposing port on the internet, without manual certificate
  • •Tailscale:private peer-to-peer network, dashboard accessible only by your devices

⚠️Don't confuse

"Dashboard doesn't open" is often "I haven't created the tunnel yet". Before assuming the Hermes bug, confirm that you have network bridged.

3

📦 .env of container vs root in Docker

Classic Docker Problem:.envfrom the hostit is notautomatically made available inside the container. Hermes reads the .envfrom inside, not from outside.

📂 Correct path

  • •The right .env is in/root/.hermes/.env inside the container
  • •On the host, mount the volume:~/.hermes:/root/.hermes
  • •Neverput API keys in.envfrom the root of the project without mounting correctly

🔍How to check

docker exec -it hermes env | grep TELEGRAM

If it doesn't return your vars, it's because the right .env isn't mounted. Edit thedocker-compose.ymland add volume.

📜 Self-writing logs

Hermes automatically writes secrets to logs in~/.hermes/logs/errors.log. You can send it to support without fear of a token leak. But confirm first — auditing logs before sharing never hurts.

TOPIC 4
4

🗜️ Automatic context compression

Symptom: agent "forgetting" something in the middle of the session. Hermescompresses long conversations automaticallybefore the context boundary. Compression is a separate LLM call — can be configured for a different provider than the main one.

⌨️ Useful commands

  • •/compress— manual compression force
  • •/usage— shows tokens used in the current session
  • •TUI status bar (v0.13.0+) displays compression count

⚙️Configure cheap model for compression

compression:
  provider: "openrouter"
  model: "google/gemini-2.5-flash"

Main model could be Claude (expensive) and compressor could be Gemini Flash (cheap). Significantly reduces cost in long sessions.

🧠LCM plugin (lossless context management)

context:
  engine: "lcm"

Alternative compressorno extra LLM call. Lossless — does not lose information due to summarization. Good when the “agent forgot” symptom is appearing too much.

TOPIC 5
5

🕐 Timezone: UTC vs local container

Docker container runs onUTC by default. Without configuring timezone, cron for "09:00" triggers at 09:00 UTC = 06:00 in Brasília. You wake up 3 hours before you expect and don't understand.

⚙️Solution 1: config.yaml

timezone: "America/Sao_Paulo"

Standard IANA string. Other examples:Europe/Lisbon, America/New_York, Asia/Tokyo.

🐳Solution 2: env var in Docker

environment:
  - TZ=America/Sao_Paulo

📅 What does this affect

  • •Timestamps in logs— all logs are in local time
  • •Cron scheduling— "09:00 every day" becomes 09:00 in your time zone
  • •Time injected into system prompt— agent knows what time it is for you

⚠️Don't forget

Did you change timezone? Restart the container. Old Crons may have the next execution calculated in the previous zone. After restart, it recalculates correctly.

TOPIC 6
6

🩺 Diagnosis via container terminal

When Telegram does not respond and you need a diagnostic port outside of it, go to the container terminal. It's Hermes' "safe mode" — the CLIhermesThere are subcommands to inspect everything.

🔧 Essential commands

# Entrar no container
docker exec -it hermes bash

# Status do gateway
hermes gateway status

# Listar crons
hermes cron list

# Ler config
hermes config get TELEGRAM_TOKEN

# Health check completo
hermes doctor

# Logs ao vivo
tail -f ~/.hermes/logs/gateway.log
tail -f ~/.hermes/logs/errors.log

📜Self-writing secrets in logs

Logs in~/.hermes/logs/apply automatic redaction: tokens, API keys and secrets become[REDACTED]. You can share with support without fear. Still, read before sending — healthy paranoia.

💡When the agent is alive but strange

Before assuming code bug, read~/.hermes/MEMORY.md e ~/.hermes/SOUL.md. 70% of "strange behavior" is an outdated fact inside. We return to this in module 6.4.
